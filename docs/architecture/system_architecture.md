# 系统总体架构

## 1. 文档目的

本文档定义 Hermes + DeepSeek BTC 合约策略辅助系统的总体架构。

本文档回答以下问题：

1. 系统整体由哪些层组成。
2. 每一层负责什么。
3. 当前阶段实现哪些能力。
4. 后续阶段预留哪些能力。
5. 哪些模块不能越界。
6. Codex 后续写代码时应如何理解系统结构。

本文档不负责：

1. 具体数据库字段设计。
2. 具体代码文件清单。
3. 具体开发步骤。
4. 具体策略算法。
5. 具体大模型 prompt。
6. 具体 Admin 页面设计。

具体需求见 `docs/requirements/`。

具体模块边界见 `docs/architecture/module_boundaries.md`。

具体数据流见 `docs/architecture/data_flow.md`。

具体开发计划见 `docs/plans/`。

## 2. 系统定位

本项目是一个面向 `BTCUSDT U 本位合约` 的低频交易辅助系统。

系统目标是辅助用户建立低频、低杠杆、可复盘、可验证、可持续优化的交易决策体系。

系统不自动交易。

系统只提供：

1. 行情数据采集。
2. 数据质量检查。
3. 基础异常告警。
4. 价格事件提醒。
5. 未来策略信号。
6. 未来大模型综合分析。
7. 未来操作建议。
8. 未来建议生命周期管理。
9. 未来复盘评估。
10. 未来人工执行记录。

系统不得实现：

1. 自动下单。
2. 自动平仓。
3. 自动调仓。
4. 自动加仓。
5. 自动减仓。
6. 自动读取账户后执行交易动作。
7. 自动根据持仓状态直接操作账户。

所有真实交易动作必须由用户手动完成。

## 5. 交易所数据接入层

交易所数据接入层负责连接 Binance U 本位合约接口。

当前主要数据源：

1. Binance U 本位合约 REST 接口。
2. Binance U 本位合约 WebSocket ticker 数据。

核心规则：

1. 4h 主 K线必须来自 Binance REST 已收盘 K线。
2. 不允许使用 WebSocket 自己拼接 4h 主 K线。
3. WebSocket 只用于最新价格监控和价格事件提醒。
4. 所有 Binance 时间字段以 UTC 为业务标准。
5. 所有 K线排序、连续性判断、策略判断必须基于 UTC。
6. PRC 时间只用于用户阅读、排查和提醒展示。

REST 适合：

1. 历史 K线回补。
2. 增量采集最新已收盘 4h K线。
3. 数据缺口修复。
4. 以交易所官方 K线为准的行情记录。

WebSocket 适合：

1. 最新价格更新。
2. 10s 价格监控。
3. 波动事件检测。
4. 接近关键价位时提醒用户。

WebSocket 不适合：

1. 作为 4h 主 K线最终数据源。
2. 替代 REST 已收盘 K线。
3. 独立生成策略建议。
4. 独立修改建议生命周期状态。

## 6. 数据采集层

数据采集层负责从交易所获取行情数据，并将可靠数据写入存储层。

当前阶段主要采集：

1. BTCUSDT U 本位合约 4h 已收盘 K线。
2. BTCUSDT 最新价格。
3. 后续可扩展 1m K线。
4. 后续可扩展 1d K线。
5. 后续可扩展标记价格 K线。
6. 后续可扩展资金费率相关数据。

4h K线采集规则：

1. 只采集已收盘 K线。
2. 不写入未收盘 K线作为正式 4h K线。
3. 入库前必须检查本批次 K线连续性。
4. 入库前必须检查与数据库最新 K线是否连续。
5. 如果发生缺口，不应继续盲目写入。
6. 如果采集失败，必须记录事件并通过 Hermes 提醒。
7. 如果 K线不连续，必须记录数据质量问题并通过 Hermes 提醒。

历史回补和增量采集是两类任务：

1. 历史回补用于补齐指定时间范围内的旧数据。
2. 增量采集用于周期性获取最新已收盘 K线。

二者可以复用 Binance REST 客户端和解析器，但必须遵守同样的数据质量规则。

## 7. 数据质量层

数据质量层负责判断行情数据是否可信。

数据质量层必须独立存在，不能散落在业务代码里。

数据质量检查至少包括：

1. K线是否已收盘。
2. K线时间是否连续。
3. K线是否重复。
4. K线 open_time 是否符合 interval。
5. K线 close_time 是否合理。
6. OHLC 价格关系是否合理。
7. 成交量是否异常。
8. 数据库最新 K线与新采集 K线是否衔接。
9. 采集失败是否被记录。
10. 缺口是否被记录。

核心原则：

1. 数据质量优先级高于尽快写入。
2. 数据异常时不得强行生成策略建议。
3. 数据异常时必须触发基础告警。
4. 数据质量异常记录必须落库。
5. 数据质量异常不能只写日志。

当前阶段数据质量异常必须通过 Hermes 提醒用户。

未来策略阶段，数据质量异常应阻止策略建议生成，或使最终建议降级为 `wait` / `stop_trading`。

## 8. 存储层

系统使用 MySQL 和 Redis。

MySQL 负责持久化数据。

Redis 负责短期缓存和事件冷却。

### 8.1 MySQL

MySQL 用于保存：

1. 4h K线数据。
2. 未来 1m K线数据。
3. 未来 1d K线数据。
4. 采集事件日志。
5. 数据质量检查结果。
6. 提醒消息记录。
7. 未来策略信号。
8. 未来策略建议。
9. 未来大模型输入输出。
10. 未来复盘结果。
11. 未来人工执行记录。

MySQL 是后续回测、复盘、策略评估的主要数据基础。

所有需要长期追溯的数据都应进入 MySQL。

### 8.2 Redis

Redis 用于保存：

1. 最新价格。
2. 短期价格状态。
3. 提醒冷却状态。
4. 任务临时状态。
5. 幂等控制辅助状态。

Redis 不作为长期行情数据库。

Redis 数据允许过期。

长期分析不能依赖 Redis。

## 9. 基础提醒层

基础提醒层负责把重要事件发送给用户。

提醒统一通过 Hermes 发送。

当前阶段提醒包括：

1. K线采集失败。
2. K线不连续。
3. 数据质量异常。
4. Binance 接口异常。
5. Redis 异常。
6. MySQL 异常。
7. 价格波动超过阈值。
8. 价格接近未来建议关键位。

基础提醒原则：

1. 基础系统提醒不得调用大模型。
2. 数据采集异常必须提醒。
3. 数据质量异常必须提醒。
4. 提醒必须落库。
5. Hermes 返回结果必须记录。
6. 提醒必须有去重和冷却机制。
7. 不能每 10s 无脑刷屏。

只有以下类型的通知后续可以调用大模型：

1. 策略分析。
2. 操作建议。
3. 建议复盘。
4. 策略总结。
5. 需要解释多策略冲突的通知。

基础异常告警不允许调用 DeepSeek、OpenAI、Grok 或其他大模型。

## 10. 未来策略信号层

策略信号层不属于当前第一阶段实现内容。

未来策略信号层负责独立运行多个策略。

策略应采用插件化、面向对象方式设计。

示例：

1. `BaseStrategy`
2. `GannStrategy`
3. `TrendStrategy`
4. `SupportResistanceStrategy`
5. `VolatilityRiskStrategy`
6. `LiquidationPressureStrategy`
7. `LeastResistanceStrategy`

每个策略应独立输出策略信号。

每个策略信号至少应包含：

1. 策略名称。
2. 策略版本。
3. 参数版本。
4. 分析周期。
5. 使用的 K线范围。
6. 策略方向。
7. 关键理由。
8. 目标区域。
9. 失效条件。
10. 风险提示。

多个策略不能混写在一个大函数里。

策略层不得直接发送最终操作建议。

策略层只输出独立信号。

清算压力与最小阻力方向策略属于未来策略信号层，不属于当前第一阶段数据采集范围。

当前阶段只需要保证数据采集、存储结构和策略插件化架构不会阻碍后续接入该类策略。

清算压力与最小阻力方向策略的核心假设是：BTCUSDT U 本位合约市场在部分阶段可能倾向于朝流动性更密集、止损更密集、清算更容易发生的方向运行。

例如：

1. 下方多头止损或清算密集时，市场可能先向下运行以清算多头。
2. 上方空头止损或清算密集时，市场可能先向上运行以清算空头。
3. 当某一方向阻力更小、流动性更集中时，行情可能优先向该方向运行。

该类策略不应被视为绝对规则。

该类策略应作为独立策略信号、风控过滤条件或策略聚合层输入参与分析。

该类策略不得单独直接生成最终操作建议。

如果该类策略与江恩策略、趋势策略或支撑压力策略发生严重分歧，最终建议可以降级为 `wait` 或 `stop_trading`，但系统必须明确告知用户是哪一种策略或风控条件导致等待或停止交易。

## 11. 未来策略聚合层

策略聚合层不属于当前第一阶段实现内容。

策略聚合层负责综合多个策略信号。

策略聚合层输入：

1. 江恩策略信号。
2. 趋势策略信号。
3. 支撑压力策略信号。
4. 波动率风控信号。
5. 未来其他策略信号。

策略聚合层输出：

1. 策略一致性分析。
2. 策略分歧分析。
3. 初步方向候选。
4. 风险提示。
5. 是否建议 `wait`。
6. 是否建议 `stop_trading`。
7. 哪些策略影响了最终倾向。

策略聚合层不能隐藏各策略观点。

即使最终建议是 `wait` 或 `stop_trading`，系统也必须能展示每个策略的简要结论和关键理由。

## 12. 未来大模型分析层

大模型分析层不属于当前第一阶段实现内容。

未来大模型分析层可以接入：

1. DeepSeek。
2. OpenAI。
3. Grok。
4. 其他大模型。

大模型的角色不是直接替代代码策略。

大模型应基于结构化输入进行综合分析。

大模型输入应包括：

1. 最新行情摘要。
2. 数据质量状态。
3. 各策略独立信号。
4. 策略聚合层摘要。
5. 风险约束。
6. 当前 active 建议状态。
7. 关键价位。
8. 复盘上下文。

大模型输出是建议候选，不是最终交易指令。

最终建议必须经过：

1. 策略聚合层。
2. 大模型分析层。
3. 最终风控硬校验。
4. 建议生命周期处理。

大模型不得直接调用交易所下单接口。

系统也不得实现交易所下单接口。

## 13. 未来建议生命周期层

建议生命周期层不属于当前第一阶段实现内容。

未来每条策略建议必须有生命周期。

系统不能每 4h 无脑生成一条全新建议。

每 4h 主评估时，系统应先检查当前 active 建议，再决定：

1. 延续当前建议。
2. 更新当前建议。
3. 完成当前建议。
4. 判定当前建议失效。
5. 判定当前建议过期。
6. 关闭当前建议。
7. 创建新建议。
8. 建议空仓等待。
9. 建议停止交易。
10. 将已结束建议放入复盘队列。
11. 执行已到期复盘。

建议生命周期必须支持：

1. `root_advice_id`
2. `parent_id`
3. `path`
4. `version_no`
5. `advice_code`
6. `status`
7. `review_status`
8. `review_due_at_utc`

当前同一 `BTCUSDT + 4h` 早期只保留一条主 active 建议链。

但主建议链必须保留其他策略和模型的重点信息。

## 14. 未来复盘评估层

复盘评估层不属于当前第一阶段实现内容。

未来复盘用于验证：

1. 策略是否有效。
2. 聚合层是否合理。
3. 大模型分析是否有帮助。
4. 最终建议是否正确。
5. 风控是否过度保守或过度激进。
6. 用户人工执行是否偏离建议。

复盘规则：

1. 建议关闭、完成、失效或过期后，不立即复盘。
2. 建议进入复盘队列。
3. 默认等待约 1 天，或 6 根 4h K线。
4. 到达 `review_due_at_utc` 后，由 4h 主调度执行复盘。
5. 每条建议链只发送一次到期复盘提醒。
6. 10s 价格监控不得触发正式复盘。

复盘评估必须支持：

1. 多空中性表达。
2. 最大有利波动。
3. 最大不利波动。
4. R 倍数。
5. 手续费和滑点。
6. 样本数量控制。
7. 市场环境分层。
8. 策略版本和参数版本对比。
9. 大模型分析效果评估。
10. 人工执行偏离评估。

复盘结果只能追加记录，不得篡改原始建议。

复盘不得自动修改策略配置或策略权重。

---

## 15. 未来人工执行记录层

人工执行记录层不属于当前第一阶段实现内容。

本项目不自动读取账户，也不自动同步真实持仓。

用户是否执行建议，需要用户手动反馈。

人工执行记录用于后续复盘。

人工执行记录可以包括：

1. 对应建议编号。
2. 执行动作。
3. 执行方向。
4. 执行价格。
5. 执行仓位。
6. 杠杆倍数。
7. 执行时间。
8. 是否按建议执行。
9. 偏离原因。
10. 减仓记录。
11. 平仓记录。
12. 用户备注。

用户确认：严格遵循系统策略时，人工执行杠杆原则上不超过 5 倍。

该规则只用于复盘人工执行风险，不用于自动调仓、自动降杠杆或自动平仓。

未来 Admin 可以用于：

3. 查看告警记录。

## 17. 部署视图

当前部署形态：

1. 一台 Ubuntu 服务器。
2. Python 项目独立虚拟环境。
3. MySQL 服务。
4. Redis 服务。
5. Hermes 服务。
6. DeepSeek API 已安装但第一阶段不参与数据采集逻辑。
7. 项目代码部署在独立目录。

推荐结构：

    /opt/hermes_btc_agent_v2
        app/
        configs/
        docs/
        migrations/
        scripts/
        tests/
        logs/
        .env
        pyproject.toml
        README.md
        AGENTS.md

Python 项目应使用独立 `.venv`。

Hermes 自身环境和本项目 Python 环境应隔离。

二者通过 HTTP Webhook 或后续约定接口协作，不应混用同一个虚拟环境。

## 18. 时间规则

系统业务时间统一使用 UTC。

UTC 用于：

1. Binance K线时间。
2. K线排序。
3. K线连续性判断。
4. 数据采集任务判断。
5. 策略分析。
6. 建议编号。
7. 建议生命周期。
8. 复盘到期时间。
9. 数据库业务字段。

PRC 时间用于：

1. 用户阅读。
2. 微信提醒展示。
3. 日志排查。
4. Admin 展示。

禁止使用 PRC 时间进行 K线连续性判断、策略判断、排序判断或建议编号。

如果需要 PRC 时间，应通过统一时间工具函数从 UTC 转换，不应在业务代码中到处手写 `+8 小时`。

## 19. 数据源权威性

4h 主 K线以 Binance REST 已收盘 K线为权威数据源。

系统不得使用以下数据替代 4h 主 K线：

1. WebSocket 拼接的 K线。
2. Redis 中的最新价格。
3. 大模型推测的数据。
4. 手工输入的数据。
5. 第三方非权威行情数据。

如果 Binance REST 数据获取失败，应记录失败并提醒用户。

如果数据不连续，应停止写入相关异常批次，并提醒用户。

数据质量问题解决前，不应继续生成依赖该数据的策略建议。

## 20. 分表原则

未来 K线应按重要周期分表。

建议方向：

1. `market_kline_1m`
2. `market_kline_4h`
3. `market_kline_1d`

4h 是当前主策略周期，优先级最高。

1m 主要用于后续复盘时分析细节，例如插针、快速波动、价格路径。

1d 主要用于后续判断大级别市场环境。

不同周期可以通过：

1. `symbol`
2. `open_time_ms`
3. `open_time_utc`

进行时间关联。

禁止依赖数据库自增 id 判断行情顺序。

行情顺序必须基于 `open_time_ms` 或 `open_time_utc`。

禁止提交：

1. `.env`
2. API Key
3. Secret
4. Token
5. 密码
6. 本地日志文件
7. 本地缓存文件
8. 虚拟环境目录
9. 生产配置明文

Codex 不得把密钥写入代码、文档、测试用例或提交记录。

## 22. 当前阶段最小闭环

当前阶段最小闭环如下：

    Binance REST
        ↓
    获取 BTCUSDT 4h 已收盘 K线
        ↓
    解析 K线
        ↓
    检查是否已收盘
        ↓
    检查本批次连续性
        ↓
    检查与数据库最新 K线衔接
        ↓
    通过则写入 MySQL
        ↓
    失败则写 collector_event_log
        ↓
    失败则写 data_quality_check
        ↓
    失败则生成 alert_message
        ↓
    通过 Hermes 发送微信提醒

实时价格提醒闭环如下：

    Binance WebSocket
        ↓
    获取最新价格
        ↓
    写入 Redis，设置 TTL
        ↓
    与上次价格比较
        ↓
    达到阈值且未处于冷却期
        ↓
    写 alert_message
        ↓
    通过 Hermes 发送微信提醒
