# 06_advice_lifecycle_requirements.md

# 策略建议生命周期需求

## 1. 文档目的

本文档定义 Hermes + DeepSeek BTC 合约策略辅助系统中“策略建议”的生命周期规则。

本文档重点约束：

1. 每 4h 如何处理当前建议。
2. 什么情况下生成新建议。
3. 什么情况下延续当前建议。
4. 什么情况下生成建议的新版本。
5. 什么情况下关闭、完成、失效或过期建议。
6. 建议之间如何建立父子关系。
7. 建议版本链如何追溯。
8. 复盘如何延迟触发。
9. 复盘如何避免重复。
10. 策略建议如何与提醒记录关联。
11. 策略建议如何与人工执行记录区分。
12. 策略建议如何保留可追溯证据链。

本文档不负责：

1. 第一阶段数据采集实现。
2. 具体策略算法实现。
3. 完整大模型分析实现。
4. 完整策略评估指标。
5. Admin 后台设计。
6. 自动交易、自动下单、自动平仓、自动调仓。
7. 账户、订单、持仓自动同步。

完整未来策略层设计由以下文档定义：

```text
docs/requirements/05_future_strategy_requirements.md
```

完整策略评估和长期盈利能力验证由以下文档定义：

```text
docs/requirements/07_strategy_evaluation_requirements.md
```

---

## 2. 核心原则

策略建议不是一次性消息。

策略建议必须有生命周期。

系统不能每 4h 简单生成一条全新建议，然后让旧建议无限堆积。

每次 4h 主周期评估时，系统必须先判断当前是否存在有效建议，再决定：

1. 延续当前建议。
2. 更新当前建议。
3. 关闭当前建议。
4. 完成当前建议。
5. 判定当前建议失效。
6. 判定当前建议过期。
7. 生成新建议。
8. 空仓等待。
9. 停止交易。
10. 将已结束建议放入复盘队列。
11. 对已到期复盘任务执行复盘。

核心原则：

```text
每 4h 必须评估
但每 4h 不一定生成新建议
建议结束后不立即复盘，而是进入复盘队列
```

## 3. strategy_advice 定位

`strategy_advice` 是未来策略系统的核心业务表。

`alert_message` 不能替代 `strategy_advice`。

二者职责不同：

```text
strategy_advice：保存策略建议本身
alert_message：保存这条建议是否发送、怎么发送、发送结果如何
```

后续当策略建议通过 Hermes 发送时，应通过：

```text
alert_message.related_type = strategy_advice
alert_message.related_id = strategy_advice.id
```

关联到对应策略建议。

要求：

1. 策略建议必须独立入库。
2. 提醒记录必须独立入库。
3. 策略建议不能只存在于微信消息中。
4. 策略建议不能只存在于大模型输出中。
5. 每条发送给用户的重要策略建议都必须能追溯到 `strategy_advice`。

---

## 4. 建议动作类型

策略建议最终动作至少包括：

```text
long
short
wait
stop_trading
```

说明：

1. `long`：建议做多。
2. `short`：建议做空。
3. `wait`：条件未到，等待更好位置。
4. `stop_trading`：风险过高、策略严重分歧或数据异常，当前停止交易。

`wait` 和 `stop_trading` 必须区分。

`wait` 表示：

```text
方向可能存在，但入场条件还没有到。
```

`stop_trading` 表示：

```text
当前不应该交易，原因可能是分歧严重、风险过高、数据异常或风控不通过。
```

停止交易是有效策略建议，不是系统失败。

---

## 5. 建议命名规则

每条策略建议必须有直观编号。

建议编号应基于 UTC 时间。

建议编号格式建议为：

```text
YYYYMMDD-SYMBOL-HH
```

示例：

```text
20260506-BTCUSDT-04
```

说明：

1. `YYYYMMDD` 使用 UTC 日期。
2. `SYMBOL` 使用交易对，例如 `BTCUSDT`。
3. `HH` 使用触发本次评估的 4h K线开盘小时，取值应为 `00`、`04`、`08`、`12`、`16`、`20`。
4. 建议编号中的时间必须与 K线业务时间标准一致，即使用 UTC。
5. PRC 时间只用于用户阅读和排查，不用于建议编号、排序、连续性判断。

如果同一个 4h 周期内产生多个版本，可使用版本号区分：

```text
20260506-BTCUSDT-04-v1
20260506-BTCUSDT-04-v2
20260506-BTCUSDT-04-v3
```

建议编号必须可读，但系统内部关联不能只依赖建议编号，应使用数据库主键和版本链字段。

---

## 6. 建议版本链

策略建议必须支持父子版本链。

当一个建议需要调整时，不应直接覆盖原建议。

正确做法是生成新版本。

示例：

```text
A-v1
  ↓
A-v2
  ↓
A-v3
  ↓
A-v4
```

数据库至少应预留以下关系能力：

```text
id
parent_id
root_advice_id
path
version_no
advice_code
```

字段含义：

1. `id`：当前建议主键。
2. `parent_id`：当前版本的上一个版本 id。
3. `root_advice_id`：整条建议链的根建议 id。
4. `path`：完整父子链路。
5. `version_no`：当前版本号。
6. `advice_code`：用户可读建议编号。

`path` 示例：

```text
/100/128/139/151
```

表示：

```text
A-v1 id = 100
A-v2 id = 128
A-v3 id = 139
A-v4 id = 151
```

要求：

1. 新版本必须能追溯父版本。
2. 任意版本必须能追溯根版本。
3. 任意版本必须能查询整条版本链。
4. 不得通过修改旧建议内容来伪装成新建议。
5. 每个版本必须保留当时的策略信号、大模型输入输出和最终建议内容。

---

## 7. 旧版本状态处理规则

当 A-v1 更新为 A-v2 时，A-v1 不应被改成 `completed` 或 `invalidated`。

A-v1 的真实状态是：

```text
已被新版本替代
```

建议使用状态：

```text
superseded
```

含义是“已被后续版本替代”。

示例：

```text
A-v1: superseded
A-v2: superseded
A-v3: superseded
A-v4: active
```

如果最终 A-v4 完成，则状态应为：

```text
A-v1: superseded
A-v2: superseded
A-v3: superseded
A-v4: completed
```

不建议把 A-v1、A-v2、A-v3 都改成 `completed`。

原因：

1. A-v1 没有独立完成。
2. A-v2 没有独立完成。
3. A-v3 没有独立完成。
4. 真正完成的是最终有效版本 A-v4。
5. 旧版本只是被替代，不能篡改生命周期事实。

整条建议链的最终结果可以通过 `root_advice_id` 或链路统计得出。

---

## 8. 建议状态

策略建议至少应支持以下状态：

```text
candidate
active
superseded
completed
invalidated
expired
closed
cancelled
```

说明：

1. `candidate`：候选建议，尚未成为正式有效建议。
2. `active`：当前有效建议。
3. `superseded`：已被新版本替代。
4. `completed`：建议目标已完成，例如到达止盈目标或按计划结束。
5. `invalidated`：建议失效，例如跌破失效条件或策略逻辑不再成立。
6. `expired`：建议超过有效期，未完成也未明确失效。
7. `closed`：系统或人工主动关闭建议。
8. `cancelled`：建议候选被取消，未进入正式生命周期。

`reviewed` 不建议作为主状态。

复盘应使用独立字段，例如：

```text
review_status
review_due_at_utc
reviewed_at_utc
reviewed_at_prc
```

原因：

1. 一个建议可以先 `completed`，再被复盘。
2. 一个建议可以先 `invalidated`，再被复盘。
3. 复盘是后续动作，不是建议本身的生命周期主状态。

---

## 9. 状态流转规则

建议状态流转应有明确规则。

允许的典型流转：

```text
candidate -> active
candidate -> cancelled

active -> superseded
active -> completed
active -> invalidated
active -> expired
active -> closed

superseded -> 不再恢复为 active

completed -> review_pending -> reviewed
invalidated -> review_pending -> reviewed
expired -> review_pending -> reviewed
closed -> review_pending -> reviewed
```

禁止的典型流转：

```text
completed -> active
invalidated -> active
expired -> active
superseded -> active
```

如果一个已结束建议需要重新启用，应创建新建议或新版本，而不是直接恢复旧状态。

---

## 10. 每 4h 主评估流程

每 4h 主周期评估时，系统应按以下顺序处理：

```text
1. 确认最新已收盘 4h K线
2. 检查数据质量
3. 查询当前 active 建议
4. 运行代码策略，生成 strategy_signal
5. 策略聚合层生成 aggregation_summary
6. 大模型综合分析层生成 llm_advice_candidate
7. 最终风控硬校验
8. 判断建议生命周期动作
9. 保存 strategy_advice 或 lifecycle_event
10. 生成 alert_message
11. 通过 Hermes 发送提醒
12. 扫描到期复盘任务
13. 对到期复盘任务执行复盘
```

如果数据质量失败，应停止后续策略建议生成。

此时应输出：

```text
数据质量异常，停止生成策略建议
```

并通过基础告警链路发送 `critical` 提醒。

---

## 11. 每 4h 可能产生的结果

每次 4h 评估可能产生以下结果之一：

```text
continue_active_advice
update_active_advice
complete_active_advice
invalidate_active_advice
expire_active_advice
close_active_advice
create_new_advice
wait_without_active_advice
stop_trading
enqueue_review
execute_due_review
```

说明：

1. `continue_active_advice`：当前建议仍有效，不生成新版本。
2. `update_active_advice`：当前建议需要调整，生成子版本。
3. `complete_active_advice`：当前建议已完成。
4. `invalidate_active_advice`：当前建议已失效。
5. `expire_active_advice`：当前建议超过有效期。
6. `close_active_advice`：主动关闭建议。
7. `create_new_advice`：无有效建议时生成新建议。
8. `wait_without_active_advice`：无有效建议，继续等待。
9. `stop_trading`：停止交易。
10. `enqueue_review`：将已结束建议放入复盘队列。
11. `execute_due_review`：执行已到期复盘任务。

---

## 12. 延续当前建议

当当前 `active` 建议仍然成立，且不需要调整点位、仓位、止损、止盈或失效条件时，应延续当前建议。

延续当前建议时：

1. 不应生成新版本。
2. 应记录一次 4h 评估事件。
3. 应说明当前建议为什么仍然有效。
4. 可以通过 Hermes 发送“当前建议继续有效”的提醒。
5. 不得重复发送旧版本完整内容造成刷屏。
6. 不得对已结束建议重复复盘。

延续提醒应包含：

1. 当前建议编号。
2. 当前建议状态。
3. 当前方向。
4. 当前关键点位。
5. 当前风险条件是否变化。
6. 下次复核时间。
7. 是否需要用户继续等待或执行。

---

## 13. 更新当前建议

当当前建议方向大体仍成立，但关键参数需要调整时，应生成新版本。

可能需要更新的内容包括：

1. 开仓区间变化。
2. 仓位比例变化。
3. 止损位置变化。
4. 止盈目标变化。
5. 减仓计划变化。
6. 保护利润规则变化。
7. 失效条件变化。
8. 风险等级变化。
9. 大模型综合分析发生明显变化。
10. 策略信号强弱发生变化。
11. 市场环境发生变化。

更新规则：

1. 新建议 `parent_id` 指向旧 active 建议。
2. 新建议 `root_advice_id` 继承根建议。
3. 新建议 `version_no = old.version_no + 1`。
4. 旧建议状态改为 `superseded`。
5. 新建议状态改为 `active`。
6. 新建议必须记录更新原因。
7. Hermes 提醒必须说明“这是旧建议的更新版本”。

禁止直接覆盖旧建议。

---

## 14. 建议更新的最小变更原则

系统不得因为轻微价格波动或无实质变化而频繁生成新版本。

只有以下情况才应生成新版本：

1. 方向变化。
2. 开仓区间发生实质变化。
3. 止损位置发生实质变化。
4. 止盈目标发生实质变化。
5. 仓位建议发生实质变化。
6. 风险等级发生变化。
7. 失效条件发生变化。
8. 策略聚合层结论发生明显变化。
9. 大模型建议候选发生明显变化。
10. 当前建议已不适合继续按原计划执行。

如果只是行情正常波动，且原建议仍然成立，应记录为 `continued`，不得生成新版本。

---

## 15. 完成当前建议

当建议达到预设目标或满足完成条件时，应将当前建议标记为 `completed`。

完成条件可以包括：

1. 到达主要止盈目标。
2. 达到计划减仓目标。
3. 建议已按计划执行完毕。
4. 策略目标已完成。
5. 系统判定当前建议无需继续跟踪。

完成后应：

1. 更新建议状态为 `completed`。
2. 记录完成时间 UTC 和 PRC。
3. 记录完成原因。
4. 记录完成时价格。
5. 将该建议放入复盘队列。
6. 计算 `review_due_at_utc`。
7. 通过 Hermes 通知用户。
8. 后续只允许复盘一次，不得重复复盘。

---

## 16. 失效当前建议

当建议的失效条件被触发时，应将当前建议标记为 `invalidated`。

失效条件可以包括：

1. 跌破做多失效位。
2. 突破做空失效位。
3. 趋势结构被破坏。
4. 江恩时间窗口失效。
5. 支撑压力结构失效。
6. 风控策略否决继续持有建议。
7. 数据质量异常导致建议不再可信。
8. 多策略结论发生严重反转。

失效后应：

1. 更新建议状态为 `invalidated`。
2. 记录失效时间 UTC 和 PRC。
3. 记录失效原因。
4. 记录失效触发价格。
5. 将该建议放入复盘队列。
6. 计算 `review_due_at_utc`。
7. 通过 Hermes 通知用户。
8. 不得继续把该建议当成 active 建议提醒。

---

## 17. 过期当前建议

建议应有有效期或最大复核周期。

如果建议在有效期内既未完成，也未明确失效，但已经不适合继续跟踪，应标记为 `expired`。

过期原因可以包括：

1. 超过建议有效时间。
2. 长时间未触发入场。
3. 市场环境已经变化。
4. 后续策略信号长期不支持该建议。
5. 价格一直未进入计划区域。

过期后应：

1. 更新建议状态为 `expired`。
2. 记录过期时间。
3. 记录过期原因。
4. 将该建议放入复盘队列。
5. 计算 `review_due_at_utc`。
6. 通过 Hermes 通知用户。
7. 后续不得重复提醒该建议仍有效。

---

## 18. 关闭当前建议

`closed` 表示系统或人工主动关闭建议。

关闭不同于完成，也不同于失效。

关闭原因可以包括：

1. 用户人工决定不再跟踪。
2. 系统进入维护状态。
3. 策略版本升级，需要关闭旧建议。
4. 数据质量问题导致建议无法继续评估。
5. 其他非市场价格触发原因。

关闭后应：

1. 更新状态为 `closed`。
2. 记录关闭原因。
3. 记录关闭时间。
4. 标记是否需要复盘。
5. 如需要复盘，则进入复盘队列。
6. 通过 Hermes 通知用户。

---

## 19. 空仓等待建议

当系统认为当前不适合开仓，但没有严重到必须停止交易时，应输出 `wait`。

`wait` 可以是：

1. 没有 active 建议时的最终建议。
2. active 建议更新后的当前动作。
3. 多策略方向存在，但入场条件未到。
4. 风险收益比暂时不合格。
5. 价格距离合理入场点过远。
6. 支撑压力位置不合适。

`wait` 提醒必须说明：

1. 当前为什么等待。
2. 等待什么条件。
3. 哪些价位值得观察。
4. 哪些条件改变后重新评估。
5. 当前是否有 active 建议。
6. 当前是否需要用户手动操作。

不得只发送：

```text
建议等待
```

而不说明原因。

---

## 20. 停止交易建议

当多策略严重分歧、风控条件不满足或数据质量异常时，应输出 `stop_trading`。

`stop_trading` 可以由以下原因触发：

1. 多策略严重分歧。
2. 风控策略否决交易。
3. 数据质量异常。
4. 止损空间过大。
5. 盈亏比不合格。
6. 波动率异常。
7. 当前行情容易诱发冲动交易。
8. 当前建议无法定义清晰失效条件。
9. 多个大模型严重分歧。
10. 代码风控硬规则拦截大模型建议。

停止交易提醒必须说明：

1. 哪些策略发生分歧。
2. 哪些风控条件不满足。
3. 哪个规则导致最终降级。
4. 大模型原本倾向是什么。
5. 最终为什么没有采纳交易方向。
6. 后续什么条件改善后可以重新评估。

停止交易是有效建议。

停止交易不是系统失败。

---

## 21. 当前有效建议唯一性

为了避免用户收到互相冲突的操作建议，系统应控制当前有效建议数量。

第一阶段策略建议阶段建议规则：

```text
同一 exchange + market_type + symbol + interval 下，只允许一个 primary active 建议链。
```

也就是说：

1. BTCUSDT 4h 当前只能有一个主有效建议链。
2. 多个策略信号可以同时存在。
3. 多个模型建议候选可以同时存在。
4. 但最终发送给用户的主操作建议只能有一个。
5. 如果新建议替代旧建议，旧建议应被 `superseded`、`closed`、`invalidated` 或 `expired`。

后续如果要支持多组并行建议，必须单独设计，例如：

1. 激进方案。
2. 保守方案。
3. 观察方案。

但早期不建议增加这种复杂度。

## 22. 主建议链与策略透明度

同一 `exchange + market_type + symbol + interval` 下，早期只保留一条主 `active` 建议链。

但只保留一条主建议链，不代表隐藏其他策略和模型的信息。

每次主建议生成、延续、更新、关闭、完成、失效或复盘时，系统仍必须展示：

1. 每个策略的简要结论。
2. 每个策略的关键理由。
3. 每个策略是否支持当前主建议。
4. 每个策略与主建议的分歧点。
5. 每个大模型的建议候选。
6. 策略聚合层为什么形成当前主建议。
7. 最终风控硬校验是否改变了大模型或聚合层倾向。

目的：

1. 用户可以独立判断每个策略表现。
2. 用户可以评估策略聚合层是否需要调整。
3. 用户可以复盘模型是否正确理解了策略信号。
4. 用户不会只收到一个黑箱式综合结论。

例如最终主建议为：

```text
wait
```

也必须展示：

```text
江恩策略：偏多，关键理由是什么
趋势策略：中性或偏空，关键理由是什么
支撑压力策略：等待回踩，关键理由是什么
风控策略：不建议追单，关键理由是什么
DeepSeek：倾向 wait，关键理由是什么
最终为什么综合为 wait
```

## 23. 建议与 K线的关联

策略建议必须能追溯到对应 K线数据。

建议不应依赖数据库 `id` 顺序判断行情连续性。

行情连续性、排序和策略判断必须以：

```text
open_time_ms
open_time_utc
```

为准。

策略建议应至少记录：

1. 使用的 K线表。
2. 主周期。
3. 使用的 K线时间范围。
4. 关键证据 K线 open_time_ms。
5. 关键证据 K线 open_time_utc。
6. 策略运行时的数据质量状态。
7. 输入数据快照引用或摘要。
8. 相关 K线数据哈希或版本标识。

原因：

如果后续回补或修正了 K线数据，系统仍应能知道某条建议当时基于什么数据做出。

后续不能只说：

```text
这条建议关联了某个 kline_id
```

还必须能知道：

```text
当时使用的是哪个时间范围的数据
当时关键证据 K线是什么
当时数据质量状态如何
```

## 24. 建议与策略信号的关联

每条最终建议必须能追溯到对应的策略信号。

未来至少应能追溯：

1. 本次使用了哪些策略。
2. 每个策略的策略名称。
3. 每个策略的版本。
4. 每个策略的参数版本。
5. 每个策略的独立结论。
6. 每个策略的理由。
7. 每个策略的风险提示。
8. 策略聚合层如何分析冲突。
9. 哪个策略或风控条件影响了最终建议。

最终建议不得只保存一个综合结论。

必须能回看：

```text
江恩策略当时怎么判断
趋势策略当时怎么判断
支撑压力策略当时怎么判断
风控策略当时怎么判断
```

## 25. 建议与大模型分析的关联

未来如果策略建议使用大模型综合分析，则必须保存大模型输入输出。

每条最终建议至少应能关联：

1. 使用的大模型供应商。
2. 使用的大模型名称。
3. 模型版本。
4. prompt 版本。
5. 输入给模型的结构化上下文。
6. 模型原始输出。
7. 模型结构化解析结果。
8. 模型建议候选。
9. 最终是否采纳模型建议。
10. 未采纳原因。

如果多个大模型参与，应能关联每个模型的独立输出和模型共识结果。

大模型输出不是最终建议。

大模型输出是建议候选。

最终建议必须经过代码风控硬校验。

## 26. 建议与 Hermes 提醒的关联

策略建议通过 Hermes 发送时，必须生成 `alert_message`。

要求：

1. 每次新建议发送必须有对应 `alert_message`。
2. 每次建议更新发送必须有对应 `alert_message`。
3. 每次建议完成、失效、关闭、过期发送必须有对应 `alert_message`。
4. 每次复盘发送必须有对应 `alert_message`。
5. `alert_message.channel_response` 必须保存 Hermes 返回结果。
6. `alert_message.related_type` 应指向 `strategy_advice` 或复盘对象。
7. `alert_message.related_id` 应指向对应业务记录。

策略建议本身不负责发送。

提醒系统负责发送、去重、冷却和发送结果记录。

## 27. 建议生命周期与人工执行状态分离

`strategy_advice` 记录的是系统给出的操作建议，不等于用户真实持仓。

由于本项目不自动下单、不自动读取账户、不自动同步持仓，系统默认不知道用户是否实际执行了建议。

因此必须区分：

1. 建议生命周期。
2. 人工执行记录。
3. 持仓观察状态。

如果用户没有记录实际执行价格、执行仓位和执行时间，系统只能基于原建议计划发送观察提醒，不能假设用户已经开仓。

涉及成本线、浮盈、浮亏、减仓、平仓、保护利润的提醒，必须基于：

1. 用户手动录入的执行记录。
2. 或原建议计划中的模拟观察条件。

系统不得在不知道用户是否持仓的情况下，直接发送“你当前持仓应该平仓”这类确定性提醒。

## 28. 人工执行反馈方式

本项目不自动下单，也不自动同步账户持仓。

用户是否按建议执行，必须由用户手动反馈。

早期可以通过 Hermes 微信命令记录人工执行结果。

示例：

```text
执行建议 20260506-BTCUSDT-04 long price=60000 size=20% leverage=3
减仓 20260506-BTCUSDT-04 price=68000 size=50% reason=到达第一目标
平仓 20260506-BTCUSDT-04 price=69200 size=100% reason=手动止盈
未执行 20260506-BTCUSDT-04 reason=没有到理想点位
```

系统收到人工执行反馈后，只能记录执行结果，用于后续提醒和复盘。

系统不得因为收到微信命令而自动下单、自动平仓、自动减仓或自动调仓。

人工执行输入必须结构化记录，不能依赖随意自然语言。

不推荐以下模糊输入：

```text
我刚刚开了点多单
```

原因：

1. 不知道是哪条建议。
2. 不知道方向。
3. 不知道价格。
4. 不知道仓位。
5. 不知道杠杆。
6. 不知道是否按建议执行。

人工执行记录至少应包含：

1. `advice_id`。
2. `advice_code`。
3. `action_type`，例如 `open`、`reduce`、`close`、`skip`。
4. `direction`，例如 `long`、`short`。
5. `price`。
6. `size_ratio`。
7. `leverage`。
8. `executed_at_utc`。
9. `executed_at_prc`。
10. `source`，例如 `weixin`、`admin`、`cli`。
11. `remark`。

人工执行记录中可以记录 `leverage`。

用户确认：正常严格执行策略时，人工杠杆原则上不超过 5 倍。

该字段只用于复盘人工执行风险，不用于系统自动调仓、自动平仓或自动读取账户状态。


后期可以通过 Admin 后台补充或修正人工执行记录。

Admin 后台记录人工执行结果，也不能变成自动下单能力。

## 29. 计划提醒与持仓提醒

系统提醒分为两类：

1. 基于建议计划的提醒。
2. 基于用户实际执行记录的提醒。

基于建议计划的提醒不要求系统知道用户是否已经开仓，只能表达：

```text
如果你已执行该建议，请检查
```

例如：

```text
当前价格已到达建议计划中的第一止盈区域。
如果你已按该建议开仓，请人工检查是否减仓或保护利润。
```

基于用户实际执行记录的提醒必须依赖用户手动录入的执行价格、执行时间、执行方向和仓位信息。

只有存在人工执行记录时，系统才能准确提醒：

1. 当前浮盈多少。
2. 当前浮亏多少。
3. 距离成本线多远。
4. 是否从最大浮盈明显回撤。
5. 是否需要保护利润。
6. 是否接近亏损。
7. 是否需要人工检查减仓或平仓。

系统不得把建议计划自动等同于用户真实持仓。

---

## 30. 平仓、减仓与保护利润提醒

策略建议生命周期不只包含开仓建议，也必须包含退出计划。

当最终建议为 `long` 或 `short` 时，系统应尽量给出：

1. 止损位置。
2. 第一止盈目标。
3. 第二止盈目标。
4. 分批减仓计划。
5. 移动止损条件。
6. 成本线保护条件。
7. 浮盈回撤提醒条件。
8. 建议失效条件。
9. 手动平仓观察条件。

系统可以通过 10s 价格监控观察当前 active 建议的关键价格条件。

允许提醒：

1. 价格接近开仓区。
2. 价格接近第一止盈区。
3. 价格接近第二止盈区。
4. 价格接近止损区。
5. 价格从浮盈区域回撤接近成本线。
6. 价格触发保护利润条件。
7. 价格接近建议失效区。
8. 行情出现剧烈反转，当前建议需要 4h 复核。

提醒内容必须明确：

1. 当前建议编号。
2. 当前建议方向。
3. 当前价格。
4. 触发的生命周期条件。
5. 原计划中的对应规则。
6. 建议用户人工检查的动作。
7. 系统不会自动平仓或自动调仓。

禁止：

1. 自动平仓。
2. 自动减仓。
3. 自动移动止损。
4. 自动调整杠杆。
5. 10s 价格监控独立生成新的策略建议。

---

## 31. 浮盈回撤与成本线保护

系统应支持对 active 建议进行浮盈回撤监控。

目的不是自动锁仓，而是提醒用户避免：

```text
盈利单变成亏损单
```

当建议已经产生明显浮盈后，系统可以根据原建议计划或人工执行记录触发提醒：

1. 是否需要部分止盈。
2. 是否需要把止损移动到成本线附近。
3. 是否需要降低仓位。
4. 是否需要等待下一根 4h K线确认。
5. 是否当前建议风险收益比已经恶化。

如果没有人工执行记录，系统只能发送基于计划的提醒。

如果存在人工执行记录，系统可以根据实际执行价格和仓位生成更准确的持仓提醒。

该提醒必须基于当前 active 建议中预先定义的保护规则，不得由 10s 价格监控临时自由生成交易建议。

---

## 32. 复盘触发规则

已结束建议应进入复盘队列。

以下状态可以触发复盘入队：

```text
completed
invalidated
expired
closed
```

复盘目标：

1. 判断建议是否有效。
2. 判断策略信号是否有效。
3. 判断大模型分析是否有效。
4. 判断最终风控是否合理。
5. 判断用户是否执行。
6. 判断用户执行是否偏离建议。
7. 为后续策略优化提供证据。

复盘不应无限堆积。

复盘不应每 4h 重复复盘同一条已关闭建议。

---

## 33. 延迟复盘触发机制

本项目以 4h 低频策略为主，建议结束后不应立即完成复盘。

当建议状态变为以下状态之一时：

```text
completed
invalidated
expired
closed
```

系统应将该建议放入复盘队列，而不是立即复盘。

建议默认复盘延迟为：

```text
review_delay_bars = 6
review_delay_interval = 4h
```

即约 1 天。

系统应计算：

```text
review_due_at_utc = advice_ended_at_utc + 24h
```

每 4h 主策略调度任务应扫描：

```text
review_status = pending
review_due_at_utc <= now_utc
```

的建议，并对到期建议执行复盘。

复盘完成后，应更新：

```text
review_status = completed
reviewed_at_utc
reviewed_at_prc
```

如果复盘失败，应记录失败原因，并允许受控重试。

10s 价格监控不得触发正式复盘。

---

## 34. 复盘去重规则

每条已结束建议默认只发送一次复盘提醒。

应使用字段控制复盘状态，例如：

```text
review_status
review_due_at_utc
reviewed_at_utc
reviewed_at_prc
review_alert_sent_at_utc
review_alert_sent_at_prc
```

`review_status` 可以包括：

```text
not_required
pending
completed
failed
```

规则：

1. `completed` 建议只复盘一次。
2. `invalidated` 建议只复盘一次。
3. `expired` 建议只复盘一次。
4. `closed` 建议是否复盘由关闭原因决定。
5. 已发送复盘提醒后，不得重复发送同一复盘提醒。
6. 如果复盘失败，可以重试，但不能重复生成多份不同结论。
7. 复盘失败也应记录原因。

用户之前确认过：

```text
最近一个已关闭建议只需要复盘一次，不要每 4h 重复发送。
```

该规则必须严格执行。

---

## 35. 每 4h 复盘范围

每次 4h 评估时，复盘范围应受控。

建议规则：

1. 复核当前 active 建议状态。
2. 扫描到期的 pending 复盘任务。
3. 对到期且未复盘的建议执行复盘。
4. 不重复复盘已经完成复盘的建议。
5. 不把所有历史建议每 4h 都重新复盘。
6. 不因为某个 active 建议长期有效，就反复发送同一个已关闭建议复盘。

正确方式：

```text
每 4h 检查是否有到期 pending review
有则处理一条或有限条
处理后标记 reviewed
下个周期不再重复
```

错误方式：

```text
每 4h 把所有历史建议重新复盘一遍
```

---

## 36. wait 与 stop_trading 的复盘

`wait` 和 `stop_trading` 也是正式策略建议，也必须支持复盘。

复盘时应评估：

1. `wait` 是否避免了低质量交易。
2. `wait` 是否错过了高质量机会。
3. `stop_trading` 是否避免了亏损。
4. `stop_trading` 是否过度保守。
5. 导致 `wait` 或 `stop_trading` 的策略分歧是否合理。
6. 风控否决是否合理。
7. 大模型是否正确理解了等待或停止交易的原因。

不能只复盘 `long` 和 `short`。

---

## 37. 生命周期状态与复盘结果分离

建议状态不等于建议质量。

例如：

1. `completed` 不一定代表策略优秀。
2. `invalidated` 不一定代表策略失败。
3. `expired` 不一定代表没有价值。
4. `wait` 可能是错过机会，也可能是正确避险。
5. `stop_trading` 可能是过度保守，也可能是避免重大亏损。

因此生命周期状态应与复盘评估结果分离。

生命周期状态描述建议发生了什么。

复盘评估结果描述建议质量如何。

复盘评估结果可以在 `07_strategy_evaluation_requirements.md` 中进一步定义。

---

## 38. 建议链复盘规则

对于 A-v1、A-v2、A-v3、A-v4 这种版本链，复盘时应区分：

1. 单个版本表现。
2. 整条建议链表现。
3. 更新是否及时。
4. 更新是否改善结果。
5. 旧版本被替代是否合理。
6. 最终有效版本是否完成或失效。

示例：

```text
A-v1: superseded
A-v2: superseded
A-v3: superseded
A-v4: completed
```

复盘时不能简单认为 A-v1、A-v2、A-v3 都 completed。

应说明：

```text
A-v1 到 A-v3 是被后续版本替代
A-v4 是最终完成版本
整条建议链最终结果为 completed
```

建议链复盘可以作为后期能力，不要求第一阶段或早期策略阶段完整实现。

但数据库和代码边界应支持未来追溯。

可预留记录：

1. 是否执行。

10. 平仓记录。
11. 最终盈亏。
12. 用户备注。

## 40. 生命周期事件记录

建议生命周期中的关键变化应记录事件。

可以预留 `strategy_advice_event` 或等价能力。

事件类型可以包括：

```text
created
activated
continued
updated
superseded
completed
invalidated
expired
closed
review_pending
review_completed
alert_sent
alert_failed
manual_execution_recorded
take_profit_zone_touched
stop_loss_zone_touched
protection_zone_touched
needs_review
```

事件记录应至少包含：

1. `advice_id`。
2. `root_advice_id`。
3. `event_type`。
4. `event_reason`。
5. `event_time_utc`。
6. `event_time_prc`。
7. `event_payload`。
8. `created_at_utc`。
9. `created_at_prc`。

早期不一定立刻实现完整事件表，但生命周期设计必须允许未来追溯。

## 41. 生命周期与时间规则

建议生命周期所有业务判断必须使用 UTC。

包括：

1. 建议编号。
2. K线周期。
3. 建议有效期。
4. 下次复核时间。
5. 复盘到期时间。
6. 复盘时间。
7. 状态流转时间。
8. 冷却时间。
9. 排序时间。

PRC 时间只用于：

1. 用户阅读。
2. 微信提醒展示。
3. 人工排查。
4. Admin 后台展示。

禁止使用 PRC 时间作为生命周期判断标准。

## 42. 生命周期与数据质量

如果数据质量异常，生命周期处理必须谨慎。

规则：

1. 数据质量异常时，不得强行生成新交易建议。
2. 数据质量异常时，不得强行更新 active 建议为新交易方向。
3. 数据质量异常时，可以将当前建议标记为需要复核。
4. 严重数据质量异常时，可以输出 `wait` 或 `stop_trading`。
5. 数据质量异常必须通过 04 文档定义的基础告警链路提醒。
6. 数据质量异常不能由大模型解释后直接覆盖。

数据质量是策略建议生命周期的前置条件。

## 43. 生命周期与 10s 价格提醒

10s 价格监控不能独立生成策略建议。

10s 价格监控可以触发：

1. 当前 active 建议接近开仓区提醒。
2. 当前 active 建议接近止损区提醒。
3. 当前 active 建议接近止盈区提醒。
4. 当前 active 建议可能失效的观察提醒。
5. 当前 active 建议触发保护利润区域提醒。
6. 当前 active 建议出现浮盈明显回撤提醒。
7. 短时间价格异常提醒。

但 10s 价格监控不得：

1. 独立生成新的 `strategy_advice`。
2. 独立关闭当前建议。
3. 独立完成当前建议。
4. 每 10s 调用大模型。
5. 每 10s 发送建议提醒。

如果 10s 价格事件表明建议可能需要变化，应标记为：

```text
needs_4h_review
```

或触发受控的复核流程。

---

## 44. 10s 价格事件与生命周期状态变更边界

10s 价格监控可以触发提醒，但默认不得直接修改 `strategy_advice` 主状态。

10s 价格监控可以标记：

1. `needs_review`。
2. `risk_triggered`。
3. `take_profit_zone_touched`。
4. `stop_loss_zone_touched`。
5. `protection_zone_touched`。

是否将建议正式改为：

```text
completed
invalidated
closed
```

应由 4h 主评估或受控复核流程确认。

如果某些规则需要 10s 级别直接触发状态变化，必须在策略规则中明确写死，不能让 10s 监控自由判断。

## 45. 低频策略建议空间要求

本项目策略建议以 4h 低频策略为主，不应生成过窄利润空间的短线建议。

当最终建议为 `long` 或 `short` 时，系统必须关注：

1. 市场阶段。
2. 目标空间。
3. 失效条件。
4. 最小预期收益。
5. 手续费。
6. 滑点。
7. 止损空间。
8. 盈亏比。
9. 是否值得交易。

如果预期利润空间不足以覆盖手续费、滑点和风险成本，系统应输出 `wait`，而不是强行给出交易方向。

建议内容可以是结构级判断，例如：

```text
当前属于熊市反弹阶段。
当前价格约 60000。
根据江恩策略，反弹目标区域可观察 68000 - 70000。
如果 4h 收盘跌破 58000，则警惕反弹结束，可能重新进入下行通道。
```

系统不得只输出过窄的开仓区间和短线目标。

## 47. 第一阶段验收标准预留

当未来进入策略建议生命周期开发时，至少应满足：

1. 每条策略建议可独立入库。
2. 每条策略建议有明确状态。
3. 每条策略建议有 UTC 建议编号。
4. 每条策略建议能记录 PRC 展示时间。
5. 每条策略建议能关联父建议。
6. 每条策略建议能追溯根建议。
7. 每条策略建议能通过 path 查询版本链。
8. 每次更新建议不覆盖旧建议。
9. 旧建议被更新后状态为 `superseded`。
10. 最终版本完成时，不篡改旧版本状态。
11. 每 4h 不强制生成新建议。
12. 每 4h 能延续、更新、关闭、完成、失效或等待。
13. 已结束建议进入复盘队列。
14. 默认复盘延迟约 1 天或 6 根 4h K线。
15. 已结束建议只复盘一次。
16. 复盘提醒不重复发送。
17. `wait` 和 `stop_trading` 也能进入复盘。
18. 生命周期状态与复盘结果分离。
19. 策略建议提醒通过 `alert_message` 关联。
20. Hermes 返回结果保存在 `alert_message.channel_response`。
21. 当前 active 建议数量受控。
22. 主建议链保留其他策略和模型的重点信息。
23. 10s 价格监控不能独立生成策略建议。
24. 10s 价格监控默认不能直接修改建议主状态。
25. 数据质量异常时不得强行生成交易建议。
26. 人工执行记录与建议生命周期分离。
27. 没有人工执行记录时，不得假设用户已经持仓。
28. 不存在自动下单、自动平仓、自动调仓能力。
