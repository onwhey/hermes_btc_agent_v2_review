# 05_future_strategy_requirements.md

# 未来策略层需求

## 1. 文档目的

本文档定义 Hermes + DeepSeek BTC 合约策略辅助系统未来策略层的设计目标、边界、模块职责和输出要求。

本文档重点约束：

1. 策略层未来要解决什么问题。
2. 策略代码如何组织。
3. 多策略如何独立运行。
4. 策略聚合层负责什么。
5. 大模型综合分析层负责什么。
6. 多个大模型如何协作。
7. 最终建议如何形成。
8. 什么情况下应该建议等待或停止交易。
9. 策略层如何保证可追溯、可复盘、可优化。
10. 策略层不得实现哪些能力。

本文档不负责：

1. 第一阶段数据采集具体实现。
2. 完整数据库字段设计。
3. 完整建议生命周期设计。
4. 完整策略评估指标设计。
5. Admin 后台设计。
6. 自动交易、自动下单、自动平仓、自动调仓。

完整建议生命周期由以下文档定义：

```text
docs/requirements/06_advice_lifecycle_requirements.md
```

完整策略评估和长期盈利能力验证由以下文档定义：

```text
docs/requirements/07_strategy_evaluation_requirements.md
```

## 2. 策略层总体定位

本项目的根本目标不是单纯减少错误操作，而是追求长期可验证的盈利能力。

策略层的职责是：

1. 基于可靠行情数据生成策略信号。
2. 对不同策略的判断进行独立记录。
3. 对多个策略之间的一致和分歧进行分析。
4. 将策略结果提供给大模型综合分析。
5. 结合大模型分析结果和硬性风控规则形成最终操作建议。
6. 在条件不满足时明确建议等待或停止交易。
7. 为后续复盘和策略优化提供可追溯依据。

策略层不是自动交易系统。

策略层只负责：

```text
分析
判断
建议
解释
复盘依据
```

策略层不得负责：

```text
自动下单
自动平仓
自动调仓
账户同步
持仓读取后自动操作
```

## 3. 当前阶段边界

第一阶段只做基础数据采集和提醒，不实现策略层。

第一阶段可以为策略层预留：

1. 清晰的目录边界。
2. 数据库预留原则。
3. K线数据质量要求。
4. `alert_message` 与未来 `strategy_advice` 的关联方向。
5. 大模型调用边界。
6. 多周期 K线采集能力。
7. 后续策略信号、模型分析、最终建议的可追溯方向。

第一阶段不实现：

1. 江恩策略。
2. 趋势策略。
3. 多策略聚合。
4. DeepSeek 策略分析。
5. 多模型协作分析。
6. 策略建议生成。
7. 策略复盘。
8. 策略回测。
9. 策略自动优化。
10. 自动交易。

## 4. 策略数据前提

策略层只能基于通过质量检查的数据运行。

策略运行前必须确认：

1. 所需 K线数据已完成采集。
2. 所需 K线数据不存在缺口。
3. 所需 K线数据已通过数据质量检查。
4. 当前数据源可信。
5. 当前时间范围没有未处理的 `critical` 数据异常。
6. Binance REST 数据源状态正常。
7. MySQL 中行情数据可查询。
8. 策略所需字段完整。

如果数据质量不满足要求，策略层不得强行生成操作建议。

此时系统应输出：

```text
数据质量异常，停止生成策略建议
```

或：

```text
数据质量异常，空仓等待
```

数据质量异常不是策略分歧，而是策略运行前置条件失败。

数据质量异常时，最终建议应倾向：

```text
wait
```

或：

```text
stop_trading
```

并明确告诉用户原因。

## 5. 主周期与辅助周期

### 5.1 4h 是主策略周期

本项目的主策略周期是 4h。

4h K线用于：

1. 主趋势判断。
2. 主要支撑压力判断。
3. 江恩时间窗口判断。
4. 主操作建议生成。
5. 建议生命周期管理。
6. 主复盘周期。

每 4h 可以进行一次主策略评估。

但每 4h 评估不等于每 4h 必须生成一条全新建议。

每次 4h 评估结果可能是：

1. 延续当前建议。
2. 更新当前建议。
3. 关闭当前建议。
4. 完成当前建议。
5. 判定当前建议失效。
6. 生成新建议。
7. 空仓等待。
8. 停止交易。
9. 触发一次复盘。

完整建议生命周期由 `06_advice_lifecycle_requirements.md` 定义。

---

### 5.2 1m K线的作用

1m K线不是主策略依据。

1m K线未来主要用于：

1. 解释价格细节。
2. 分析是否插针。
3. 分析短时间剧烈波动。
4. 复盘某条建议触发过程。
5. 观察入场点附近的微观价格行为。
6. 分析止损是否被瞬间扫掉。
7. 辅助判断提醒事件发生时的具体行情过程。

1m K线不得单独推翻 4h 主策略。

1m K线不得独立生成主操作建议。

---

### 5.3 1d K线的作用

1d K线未来主要用于：

1. 大级别趋势背景。
2. 市场环境判断。
3. 牛市、熊市、震荡环境识别。
4. 关键大级别压力和支撑判断。
5. 过滤 4h 策略信号。
6. 降低逆大趋势交易风险。

1d K线不直接替代 4h 主策略周期。

---

### 5.4 10s 价格监控的作用

10s 价格监控是事件提醒工具，不是策略生成工具。

10s 价格监控用于：

1. 发现短时间价格剧烈波动。
2. 发现价格接近关键位。
3. 发现价格触发已有计划中的观察条件。
4. 发现价格接近止损、止盈、失效区。
5. 触发 Hermes 微信提醒。

10s 价格监控不得：

1. 独立生成策略建议。
2. 每 10s 调用 DeepSeek。
3. 每 10s 调用任何大模型。
4. 每 10s 推翻 4h 主策略。
5. 每 10s 发送提醒。
6. 伪装成操作建议。

10s 是检查频率，不是提醒频率。

## 6. 策略代码组织要求

未来策略层必须采用清晰、可扩展、可测试的结构。

策略之间不能混写在一个大函数里。

应采用类似以下直观结构：

```text
BaseStrategy
GannStrategy
TrendStrategy
SupportResistanceStrategy
VolatilityRiskStrategy
```

要求：

1. 一个策略一个独立文件或独立模块。
2. 一个策略一个独立类。
3. 所有策略应遵循统一基础接口。
4. 策略之间不得直接互相修改结果。
5. 策略输出必须结构化。
6. 策略运行结果必须可以单独保存。
7. 后续新增策略不得破坏已有策略接口。

示例逻辑结构：

```text
BaseStrategy
  ├── GannStrategy
  ├── TrendStrategy
  ├── SupportResistanceStrategy
  ├── VolatilityRiskStrategy
  └── FutureOtherStrategy
```

其中：

1. `BaseStrategy` 表示基础策略接口。
2. `GannStrategy` 表示江恩策略。
3. `TrendStrategy` 表示趋势策略。
4. `SupportResistanceStrategy` 表示支撑压力策略。
5. `VolatilityRiskStrategy` 表示波动率和风险过滤策略。

具体代码命名可以按照 Python 项目规范调整，但逻辑上必须保持“一个策略一个独立模块”的边界。

## 7. 策略基础接口要求

每个策略至少应具备以下能力：

1. 接收标准化市场数据。
2. 校验输入数据是否满足策略运行要求。
3. 输出策略方向判断。
4. 输出关键理由。
5. 输出关键证据。
6. 输出风险提示。
7. 输出无效条件。
8. 输出置信度或强弱等级。
9. 输出是否建议交易。
10. 输出是否建议等待。
11. 输出是否建议停止交易。
12. 输出策略版本。
13. 输出策略参数版本。
14. 输出策略运行时间。
15. 输出使用的数据范围。

策略不应直接负责：

1. Hermes 发送。
2. 微信提醒。
3. 最终综合建议。
4. 自动下单。
5. 修改其他策略结果。
6. 调用交易所下单接口。
7. 直接调用大模型生成最终建议。

策略只输出 `strategy_signal`。

---

## 8. strategy_signal 输出要求

每个独立策略运行后，应输出结构化 `strategy_signal`。

`strategy_signal` 至少应包含：

1. 策略名称。
2. 策略版本。
3. 策略参数版本。
4. 交易所。
5. 市场类型。
6. 交易对。
7. 主周期。
8. 使用的数据范围。
9. 方向判断。
10. 策略结论。
11. 判断理由。
12. 关键证据。
13. 风险条件。
14. 无效条件。
15. 建议动作。
16. 置信度或强弱等级。
17. 是否允许交易。
18. 是否建议等待。
19. 是否建议停止交易。
20. 运行时间。
21. 输入数据快照引用。

方向判断可以包括：

```text
bullish
bearish
neutral
conflict
invalid
```

建议动作可以包括：

```text
long
short
wait
stop_trading
no_signal
```

其中：

1. `long` 表示策略倾向做多。
2. `short` 表示策略倾向做空。
3. `wait` 表示条件未到，等待。
4. `stop_trading` 表示风险过高或条件冲突，停止交易。
5. `no_signal` 表示策略没有产生有效信号。

## 9. 多策略运行方式

多策略必须先独立运行，再统一聚合。

禁止多个策略混在一起直接生成最终建议。

正确流程：

```text
1. GannStrategy 独立运行
2. TrendStrategy 独立运行
3. SupportResistanceStrategy 独立运行
4. VolatilityRiskStrategy 独立运行
5. 每个策略分别生成 strategy_signal
6. 保存每个 strategy_signal
7. 策略聚合层分析策略一致性和冲突
8. 大模型综合分析层基于完整材料生成建议候选
9. 最终风控校验层进行硬性校验
10. 生成最终 strategy_advice
11. 通过 Hermes 发送给用户
```

错误流程：

```text
多个策略混在一个函数里直接输出买卖建议
```

错误流程：

```text
每个策略各自直接给用户发送最终操作建议
```

错误流程：

```text
直接把行情丢给大模型，让大模型绕过代码策略自由判断买卖
```

---

## 10. 策略聚合层定位

策略聚合层不是一个具体交易策略。

策略聚合层是：

```text
证据整理层
冲突分析层
风控预判层
大模型输入整理层
```

策略聚合层不应直接成为最终裁判。

策略聚合层的职责是把各个独立策略的结果整理成结构化材料，供大模型综合分析层和最终风控校验层使用。

策略聚合层可以输出：

```text
aggregation_summary
preliminary_action
risk_flags
conflict_flags
candidate_actions
excluded_actions
```

但 `preliminary_action` 只是初步倾向，不是最终建议。

最终建议必须经过：

```text
大模型综合分析
模型共识或分歧比较
最终风控硬校验
建议生命周期检查
```

---

## 11. 策略聚合层要求

策略聚合层至少应判断：

1. 多个策略是否方向一致。
2. 多个策略是否严重分歧。
3. 哪些策略支持做多。
4. 哪些策略支持做空。
5. 哪些策略建议等待。
6. 哪些策略建议停止交易。
7. 是否存在风险过滤不通过。
8. 是否存在数据质量不通过。
9. 是否存在盈亏比不合格。
10. 是否存在止损空间过大。
11. 是否需要空仓等待。
12. 是否允许形成操作建议候选。
13. 哪些动作应被排除。
14. 哪些问题需要交给大模型重点分析。

聚合层不应该简单投票。

例如：

```text
3 个策略看多，1 个风控策略禁止交易
```

不能简单得出：

```text
多数看多，所以做多
```

正确处理可能是：

```text
方向偏多，但风控不通过，初步动作倾向 wait 或 stop_trading。
```

---

## 12. 策略聚合透明度要求

策略聚合层不得只输出一个不透明的最终结论。

如果聚合层初步倾向为 `wait` 或 `stop_trading`，系统必须明确说明：

1. 哪些策略支持交易。
2. 哪些策略反对交易。
3. 哪些策略建议等待。
4. 哪个风控条件没有通过。
5. 是否存在严重策略分歧。
6. 为什么暂时没有给出开仓建议。
7. 用户后续需要等待什么条件改善。

波动率与风控策略可以对最终建议产生否决效果，使最终建议变为：

```text
wait
```

或：

```text
stop_trading
```

但这种否决必须可解释、可追溯、可复盘。

系统不能只给用户一个黑箱结论。

用户必须拥有完整知情权。

---

## 13. 多策略分歧处理规则

多策略严重分歧时，不应强行给出交易方向。

严重分歧包括但不限于：

1. 江恩策略看多，趋势策略看空。
2. 趋势策略看多，但支撑压力策略没有合格入场点。
3. 多个策略方向相反。
4. 风控策略提示止损空间过大。
5. 波动率异常导致入场风险过高。
6. 数据质量不满足策略运行要求。
7. 盈亏比不合格。
8. 当前价格距离关键位过远，追单风险高。
9. 策略信号互相冲突且无法解释。

严重分歧时，最终建议应倾向：

```text
stop_trading
```

或：

```text
wait
```

停止交易不是系统失败。

停止交易是一种有效策略。

系统必须能够解释为什么停止交易。

---

## 14. 大模型综合分析层定位

大模型在未来策略系统中不只是解释器。

大模型不是简单润色策略聚合层结果。

大模型应作为：

```text
综合分析员
风险质疑者
冲突解释者
建议候选生成者
用户可读表达组织者
```

大模型应接收完整的结构化策略分析上下文，包括：

1. 每个独立策略的 `strategy_signal`。
2. 每个策略的方向、理由、证据、风险、失效条件。
3. 策略聚合层的一致性与分歧分析。
4. 风控策略结果。
5. 数据质量状态。
6. 当前有效建议及其生命周期状态。
7. 当前市场摘要。
8. 需要大模型重点分析的问题。

大模型不得只接收一个简单的多空结果。

如果只把多空结果交给大模型，大模型只能做语言解释，不能发挥综合分析价值。

---

## 15. 大模型职责

大模型的合理职责包括：

1. 综合分析多个策略信号。
2. 识别策略之间的逻辑冲突。
3. 质疑聚合层初步结论。
4. 补充风险解释。
5. 判断是否应 `long`、`short`、`wait` 或 `stop_trading`。
6. 输出最终建议候选。
7. 说明为什么不是其他动作。
8. 明确后续需要等待或观察的条件。
9. 对停止交易原因进行解释。
10. 对策略复盘进行总结。
11. 辅助发现策略优化方向。

大模型不得负责：

1. 直接从未经校验的原始行情中随意生成买卖建议。
2. 绕过代码策略直接输出最终建议。
3. 绕过数据质量检查。
4. 绕过风险过滤。
5. 自动下单。
6. 自动平仓。
7. 自动调仓。
8. 直接修改生产策略代码并上线。

---

## 16. 大模型只能输出建议候选

大模型综合分析层输出的是：

```text
llm_advice_candidate
```

不是最终：

```text
strategy_advice
```

最终 `strategy_advice` 必须经过代码层最终校验，包括：

1. 数据质量校验。
2. 风控硬规则校验。
3. 策略分歧校验。
4. 止损与盈亏比校验。
5. 建议生命周期状态校验。
6. 提醒发送前格式校验。

如果大模型建议与硬性风控规则冲突，必须以风控规则为准。

最终建议应降级为：

```text
wait
```

或：

```text
stop_trading
```

并明确告诉用户：

1. 大模型倾向是什么。
2. 最终建议为什么没有采纳大模型倾向。
3. 哪条风控规则产生了否决效果。
4. 后续什么条件改善后可以重新评估。

---

## 17. 大模型输入要求

大模型输入必须优先使用结构化 JSON 或等价结构化数据。

建议命名为：

```text
llm_analysis_context
```

输入内容应包括：

```text
market_context
data_quality_status
strategy_signals
aggregation_summary
risk_checks
active_advice_context
required_output_schema
```

至少应包含：

1. 交易所。
2. 市场类型。
3. 交易对。
4. 主周期。
5. 当前评估时间 UTC。
6. 当前有效建议。
7. 当前建议版本链。
8. 数据质量状态。
9. 最近 N 根 4h K线摘要。
10. 最近关键高点。
11. 最近关键低点。
12. 当前价格。
13. 当前 1d 趋势背景。
14. 近期波动率。
15. 是否出现插针。
16. 是否接近支撑。
17. 是否接近压力。
18. 每个策略的独立信号。
19. 策略聚合层冲突分析。
20. 风控检查结果。
21. 希望大模型重点回答的问题。
22. 要求大模型输出的结构化格式。

禁止只用自然语言描述行情后让大模型自由判断。

---

## 18. 大模型输出要求

大模型输出必须尽量结构化。

建议输出命名为：

```text
llm_advice_candidate
```

至少应包含：

1. `candidate_action`
2. `direction`
3. `confidence_level`
4. `main_reasons`
5. `main_risks`
6. `disagreement_analysis`
7. `invalidation_conditions`
8. `wait_conditions`
9. `stop_trading_reason`
10. `whether_agree_with_aggregation`
11. `whether_risk_rules_should_override`
12. `human_readable_summary`

其中 `candidate_action` 至少支持：

```text
long
short
wait
stop_trading
```

大模型输出必须保存原始内容和结构化解析结果。

如果模型输出无法解析，系统不得直接把无法解析内容作为最终建议发送给用户。

---

## 19. 大模型输入输出保存要求

未来策略层必须保存大模型分析过程。

至少包括：

1. `model_provider`
2. `model_name`
3. `model_version`
4. `prompt_version`
5. 输入给模型的结构化上下文。
6. 模型原始输出。
7. 模型结构化解析结果。
8. 模型建议候选。
9. 模型风险提示。
10. 最终建议是否采纳该模型观点。
11. 如果未采纳，未采纳原因。
12. 模型调用时间。
13. 模型调用状态。
14. 模型调用失败原因。

如果不保存大模型输入输出，后续无法判断：

1. 是策略错了。
2. 是模型解释错了。
3. 是聚合层错了。
4. 是最终风控拦截错了。
5. 是用户执行偏离了建议。

因此，大模型输入输出保存是未来策略复盘能力的前置条件。

---

## 20. 多模型协作要求

后续如接入 DeepSeek、OpenAI、Grok 或其他多个大模型，多个模型应基于同一份结构化策略分析上下文独立输出判断。

多模型不应只接收一个简单的多空结果。

每个模型至少应输出：

1. 倾向动作。
2. 主要理由。
3. 主要风险。
4. 是否同意聚合层初步结论。
5. 是否建议降级为 `wait`。
6. 是否建议 `stop_trading`。
7. 哪些条件改变后可以重新评估。

系统应对多个模型输出进行结构化比较。

如果多个模型严重分歧，最终建议应倾向：

```text
wait
```

或：

```text
stop_trading
```

并明确告诉用户分歧原因。

---

## 21. 多模型不是自由聊天

多模型协作默认采用：

```text
独立分析 + 结构化比较
```

而不是：

```text
多个模型自由聊天
```

多个大模型应接收同一份结构化策略分析上下文，分别输出结构化分析结果。

系统再对多个模型输出进行比较，包括：

1. 是否方向一致。
2. 是否都建议等待。
3. 是否有模型建议停止交易。
4. 分歧点在哪里。
5. 哪个模型识别了更多风险。
6. 是否需要降级为 `wait` 或 `stop_trading`。

第一阶段和早期策略阶段不实现模型之间自由对话。

---

## 22. 模型共识与分歧比较

未来如果接入多个大模型，系统应有模型共识与分歧比较层。

该层可以输出：

```text
model_consensus_result
```

它应分析：

1. 多个模型是否一致。
2. 多个模型是否方向相反。
3. 哪些模型支持 `long`。
4. 哪些模型支持 `short`。
5. 哪些模型建议 `wait`。
6. 哪些模型建议 `stop_trading`。
7. 分歧是否严重。
8. 分歧原因是什么。
9. 是否需要人工确认。
10. 是否需要降级为 `wait` 或 `stop_trading`。

模型共识层不得绕过最终风控硬校验。

## 23. 最终风控硬校验

最终建议生成前，代码层必须执行硬性风控校验。

校验内容至少包括：

1. 数据质量是否通过。
2. 是否存在未处理的 `critical` 数据异常。
3. 是否有明确止损。
4. 盈亏比是否合格。
5. 是否违反风控策略否决。
6. 是否存在严重策略分歧。
7. 是否缺少入场点。
8. 是否需要高杠杆才能成立。
9. 当前建议是否与建议生命周期冲突。
10. 当前建议是否缺少失效条件。

如果硬规则不通过，最终建议必须被降级为：

```text
wait
```

或：

```text
stop_trading
```

同时必须告诉用户降级原因。

---

## 24. 最终建议生成结构

未来最终策略建议生成流程应为：

```text
第一层：独立代码策略层
GannStrategy
TrendStrategy
SupportResistanceStrategy
VolatilityRiskStrategy

第二层：策略聚合层
整理每个策略信号
分析一致与冲突
生成 aggregation_summary
生成 preliminary_action
标记风险否决条件

第三层：大模型综合分析层
DeepSeek / OpenAI / Grok 等模型读取完整策略材料包
输出各自结构化 llm_advice_candidate

第四层：模型共识与分歧比较层
比较多个模型结果
识别模型共识与模型冲突

第五层：最终风控硬校验层
执行数据质量、风控、止损、盈亏比、生命周期等硬规则

第六层：最终建议层
生成 strategy_advice
保存证据链
通过 Hermes 发送
```

关键边界：

```text
策略聚合层不是最终裁判。
大模型不是最终裁判。
最终建议必须经过代码风控硬校验。
```

## 25. 策略建议展示要求

未来策略建议提醒不是只给一个综合结论。

用户端应看到：

1. 每种策略的简要结论。
2. 每种策略为什么这么判断。
3. 策略之间是否一致。
4. 如果存在分歧，分歧点在哪里。
5. 分歧是否严重。
6. 风控条件是否通过。
7. 大模型综合分析结论。
8. 如果多个大模型参与，模型之间是否一致。
9. 最终综合建议。
10. 如果建议停止交易，说明停止交易原因。
11. 如果建议空仓等待，说明等待什么条件。
12. 如果建议开仓，说明关键点位、仓位比例和失效条件。

建议展示结构应类似：

```text
一、各策略独立判断

1. 江恩策略
方向：偏多 / 偏空 / 观望
理由：......

2. 趋势策略
方向：偏多 / 偏空 / 观望
理由：......

3. 支撑压力策略
方向：等待回踩 / 等待突破 / 不追单
理由：......

4. 波动率风控策略
方向：可交易 / 降低仓位 / 禁止交易
理由：......

二、策略一致性与分歧

说明哪些策略一致，哪些策略冲突。

三、风险检查

说明止损空间、盈亏比、波动率、数据质量等条件是否满足。

四、大模型综合分析

说明大模型如何看待策略信号、风险和分歧。

五、最终建议

long / short / wait / stop_trading

六、操作要点

开仓区间、仓位比例、止损、止盈、失效条件、复核时间。
```

---

## 26. 最终建议动作

未来最终建议动作至少包括：

```text
long
short
wait
stop_trading
```

说明：

1. `long`：建议做多。
2. `short`：建议做空。
3. `wait`：条件未到，等待更好位置。
4. `stop_trading`：多策略严重分歧或风险条件不满足，当前停止交易。

`wait` 与 `stop_trading` 必须区分。

`wait` 表示：

```text
方向可能存在，但入场条件还没有到。
```

`stop_trading` 表示：

```text
当前不应该交易，原因可能是分歧严重、风险过高、数据异常或风控不通过。
```

---

## 27. 操作建议内容要求

当最终建议为 `long` 或 `short` 时，必须尽量包含：

1. 多空方向。
2. 开仓区间。
3. 开仓比例。
4. 分批计划。
5. 止损位置。
6. 止盈目标。
7. 减仓条件。
8. 加仓条件。
9. 失效条件。
10. 最大可接受亏损。
11. 风险收益比。
12. 当前建议有效期。
13. 下次复核时间。
14. 不应该交易的条件。

如果无法给出清晰止损或风险收益比不合理，应倾向：

```text
wait
```

或：

```text
stop_trading
```

而不是强行给出方向。

## 28. 风控优先原则

合约交易中，风控优先级高于方向判断。

即使方向判断正确，如果风险条件不满足，也不应建议交易。

以下情况应优先考虑 `wait` 或 `stop_trading`：

1. 止损空间过大。
2. 盈亏比不合格。
3. 价格已经远离合理入场点。
4. 多策略严重分歧。
5. 波动率异常。
6. 数据质量异常。
7. 关键位置附近假突破风险高。
8. 当前建议无法定义明确失效条件。
9. 当前交易需要过高杠杆才能满足预期收益。
10. 用户容易因该类行情冲动交易。

系统不应为了“每 4h 都给建议”而强行生成交易建议。

---

## 29. 江恩策略要求

江恩策略是未来重点策略之一。

江恩策略可以关注：

1. 时间周期。
2. 变盘时间。
3. 关键价格位。
4. 时间与价格共振。
5. 历史高低点关系。
6. 重要角度或比例关系。
7. 趋势转折风险。

江恩策略必须结构化输出。

江恩策略不得直接单独形成最终建议。

江恩策略输出应进入 `strategy_signal`，再由聚合层、大模型综合分析层和最终风控校验层共同处理。

---

## 30. 趋势策略要求

趋势策略用于判断当前行情主方向。

趋势策略可以关注：

1. 高低点结构。
2. 趋势延续。
3. 趋势破坏。
4. 均线方向。
5. 突破与回踩。
6. 趋势强弱。
7. 多空结构切换。

趋势策略输出应与江恩策略独立保存。

趋势策略不得与江恩策略混写。

---

## 31. 支撑压力策略要求

支撑压力策略用于判断关键价格区域。

支撑压力策略可以关注：

1. 前高。
2. 前低。
3. 密集成交区域。
4. 突破位。
5. 回踩位。
6. 压力区。
7. 支撑区。
8. 假突破风险。
9. 入场位置是否合理。

支撑压力策略重点解决：

```text
在哪里交易
```

而不是单独解决：

```text
一定做多还是一定做空
```

---

## 32. 波动率与风控策略要求

波动率与风控策略用于过滤高风险交易。

该策略可以关注：

1. 当前波动率是否异常。
2. 止损空间是否过大。
3. 价格是否发生插针。
4. 当前行情是否适合开仓。
5. 是否需要降低仓位。
6. 是否应停止交易。
7. 交易风险是否超过可接受范围。

波动率与风控策略具有否决权。

如果该策略判断风险过高，最终建议不得简单因为其他策略看多或看空就强行交易。

但否决必须明确告诉用户原因。

例如：

```text
最终建议：wait
原因：江恩和趋势策略偏多，但当前价格距离合理止损位过远，盈亏比不合格。
```

或：

```text
最终建议：stop_trading
原因：趋势策略与江恩策略方向冲突，同时波动率异常，风控策略禁止交易。
```

## 32.1. 清算压力与最小阻力方向策略预留

后续系统应预留清算压力与最小阻力方向策略能力。

该策略的核心假设是：BTCUSDT U 本位合约市场在部分阶段可能倾向于朝流动性更密集、止损更密集、清算更容易发生的方向运行。

例如：

1. 下方多头止损或清算密集时，市场可能先向下运行以清算多头。
2. 上方空头止损或清算密集时，市场可能先向上运行以清算空头。
3. 当某一方向阻力更小、流动性更集中时，行情可能优先向该方向运行。

该策略不应被视为绝对规则。

该策略应作为独立策略信号、风控过滤条件或策略聚合层输入参与分析。

后续可实现为：

1. `LiquidationPressureStrategy`
2. `LeastResistanceStrategy`

该策略可以输出：

1. 当前更容易清算多头还是空头。
2. 上方是否存在潜在空头清算压力。
3. 下方是否存在潜在多头清算压力。
4. 当前方向是否属于阻力较小方向。
5. 当前是否存在诱多或诱空风险。
6. 当前是否应该降低仓位。
7. 当前是否应该 `wait`。
8. 当前是否应该 `stop_trading`。

该策略后续可能依赖以下数据：

1. 4h K线。
2. 1m K线。
3. 标记价格。
4. 指数价格。
5. 资金费率。
6. 持仓量。
7. 多空账户比例。
8. 多空持仓比例。
9. 主动买入量和主动卖出量。
10. 强平订单或清算事件数据。

该策略不得单独直接生成最终操作建议。

该策略输出应先保存为独立策略信号，再进入策略聚合层和大模型分析层。

如果该策略与江恩策略、趋势策略或支撑压力策略发生严重分歧，最终建议可以降级为 `wait` 或 `stop_trading`，但必须明确告知用户是哪一种策略或风控条件导致等待或停止交易。

## 33. 策略版本与参数版本要求

每个策略必须有明确版本。

策略版本用于：

1. 回测。
2. 复盘。
3. 策略效果对比。
4. 判断某次建议使用了哪个策略版本。
5. 后续策略优化。
6. 避免修改策略后无法解释历史建议。

策略输出必须记录：

```text
strategy_name
strategy_version
parameter_version
```

策略代码变化应升级：

```text
strategy_version
```

策略参数变化应升级：

```text
parameter_version
```

示例：

```text
strategy_name = GannStrategy
strategy_version = v1
parameter_version = p20260506
```

不允许修改策略逻辑后仍使用完全相同版本号，导致历史复盘混乱。

不允许修改策略参数后仍无法区分历史运行结果。

## 33.1 策略独立配置文件要求

未来每个策略应支持独立配置文件。

策略参数不得硬编码在业务代码中。

建议配置目录：

```text
configs/strategies/gann.yaml
configs/strategies/trend.yaml
configs/strategies/support_resistance.yaml
configs/strategies/risk.yaml
```

每个策略配置文件至少应支持：

1. `strategy_name`；
2. `strategy_version`；
3. `parameter_version`；
4. 主周期；
5. 辅助周期；
6. 入场区域计算方式；
7. 目标空间计算方式；
8. 最小预期收益；
9. 最小盈亏比；
10. 手续费参数；
11. 滑点参数；
12. 止损或失效条件；
13. 失效条件确认方式，例如 4h 收盘确认；
14. 减仓规则；
15. 保护利润规则；
16. 复盘延迟参数；
17. 风控阈值；
18. 策略启用或禁用开关。

策略代码变化应升级：

```text
strategy_version
```

策略参数变化应升级：

```text
parameter_version
```

后续复盘时，系统必须能知道某次策略运行使用的是哪一个策略版本和哪一组参数版本。

禁止修改策略参数后仍使用同一个 `parameter_version`，导致历史结果无法解释。

策略配置文件应被纳入版本管理。

但配置文件中不得保存密钥、账户信息、交易所 API Key、大模型 API Key 或其他敏感信息。

## 33.2 低频策略参数原则

本项目以 4h 低频策略为主，不应默认生成过窄的短线开仓区间和过小目标空间。

策略配置应支持较大级别结构判断。

例如：

```text
当前属于熊市反弹阶段。
当前价格约 60000。
根据江恩策略，反弹目标区域观察 68000 - 70000。
如果 4h 收盘跌破 58000，则警惕反弹结束，可能重新进入下行通道。
```

未来策略建议不应只关注：

```text
某个极窄开仓区间
某个很小止盈空间
短时间价格噪音
```

而应重点关注：

1. 当前市场阶段；
2. 当前结构属于趋势、震荡、反弹还是下跌延续；
3. 主要目标区域；
4. 关键失效条件；
5. 预期收益空间；
6. 止损或失效空间；
7. 盈亏比；
8. 手续费；
9. 滑点；
10. 是否值得交易。

如果预期利润空间不足以覆盖手续费、滑点和风险成本，系统应输出：

```text
wait
```

而不是强行给出：

```text
long
```

或：

```text
short
```

策略参数应支持后续调整。

例如：

```text
最小预期涨跌幅
最小盈亏比
入场区间宽度
目标空间阈值
止损或失效条件
手续费过滤
滑点过滤
复盘延迟时间
```

这些参数应优先从策略独立配置文件读取，不应写死在策略业务代码中。

如果用户后续认为某个策略过于激进、过于保守、入场区间过窄、目标空间过小，应可以通过调整策略配置文件进行优化。

每次参数调整都必须升级 `parameter_version`，否则后续无法判断历史建议到底使用了哪一组参数。

## 34. 策略输入快照要求

策略运行时必须记录其使用的数据范围和关键输入。

后续不能只依赖当前行情主表复盘。

策略运行记录未来至少应能追溯：

1. 使用了哪些 K线。
2. K线时间范围。
3. 关键证据 K线。
4. 市场环境快照。
5. 策略版本。
6. 策略参数版本。
7. 输入参数。
8. 数据质量状态。
9. 策略聚合层输入。
10. 大模型输入。
11. 大模型输出。
12. 最终建议。

第一阶段不实现完整快照表，但未来策略层必须支持该能力。

---

## 35. 市场环境分类要求

未来策略评估需要区分不同市场环境。

系统后续应能识别或记录：

1. 牛市。
2. 熊市。
3. 震荡。
4. 高波动。
5. 低波动。
6. 趋势行情。
7. 区间行情。
8. 资金费率异常环境。
9. 插针或极端波动环境。

原因：

同一个策略在不同市场环境下表现可能完全不同。

例如：

1. 江恩策略可能在时间窗口附近更有效。
2. 趋势策略可能在强趋势中更有效。
3. 支撑压力策略可能在震荡行情中更有效。
4. 波动率风控策略可能在极端行情中更重要。

具体评估指标由 `07_strategy_evaluation_requirements.md` 定义。

---

## 36. 策略不得使用的数据

策略层不得使用以下不可靠或越界数据作为主判断依据：

1. 未收盘 4h K线。
2. 未通过质量检查的 K线。
3. 存在缺口但未通过 Binance REST 回补的数据。
4. WebSocket 自行拼接的 4h K线。
5. Redis 中已过期或不完整的短期状态。
6. 未记录来源的人工数据。
7. 未保存快照的大模型自由判断。
8. 未经过结构化的主观描述。

如果未来使用人工标注数据，只能用于行情环境标签、复盘标签或策略研究标签，必须明确记录来源和时间。

人工标注数据不得用于替代、覆盖或修正正式 K线表中的 OHLCV 等核心行情字段。

## 37. 策略与提醒系统关系

策略层不直接发送提醒。

策略层输出：

```text
strategy_signal
aggregation_summary
llm_advice_candidate
model_consensus_result
strategy_advice
```

提醒系统负责：

```text
alert_message
Hermes 发送
channel_response
去重
冷却
发送失败记录
```

策略建议提醒必须通过 Hermes。

策略建议提醒可以使用大模型生成解释文本，但最终发送仍由代码调用 Hermes 完成。

基础系统提醒不得调用大模型。

---

## 38. 策略与人工执行关系

本项目只给用户操作建议。

用户是否执行、如何执行、执行多少仓位，由用户手动决定。

未来系统可以记录人工执行结果，用于复盘和评估。

但系统不得：

1. 自动读取账户后决定下单。
2. 自动下单。
3. 自动平仓。
4. 自动调仓。
5. 自动调整杠杆。
6. 自动追踪持仓后发出强制操作。

后续可以记录：

1. 用户是否执行建议。
2. 实际执行方向。
3. 实际执行价格。
4. 实际执行仓位。
5. 是否偏离建议。
6. 偏离原因。
7. 偏离后的结果。

这些内容主要由后续复盘和评估文档定义。

---

## 39. 策略优化边界

系统未来可以根据长期复盘结果辅助优化策略。

但策略优化必须基于可验证数据。

禁止：

1. 因为单次亏损就随意修改策略。
2. 因为单次盈利就认为策略有效。
3. 只看胜率，不看盈亏比。
4. 只看盈利，不看最大回撤。
5. 只看方向正确，不看是否可执行。
6. 使用未来数据污染历史回测。
7. 让大模型直接修改核心策略代码并上线。
8. 未经评估直接替换策略版本。

策略优化应基于：

1. 足够数量的建议样本。
2. 盈亏比。
3. 期望值。
4. 最大浮盈。
5. 最大浮亏。
6. 最大回撤。
7. 连续亏损。
8. 不同市场环境表现。
9. 多策略一致和分歧时的表现。
10. 用户执行偏差。

详细评估规则由 `07_strategy_evaluation_requirements.md` 定义。

---

## 40. 大模型不得直接修改策略代码

大模型可以提出策略优化建议，但不得直接修改并上线核心策略代码。

策略优化必须经过：

1. 人工确认。
2. 新策略版本定义。
3. 回测或历史验证。
4. 独立分支开发。
5. 代码审查。
6. 版本记录。
7. 上线后持续复盘。

禁止因为大模型一次分析结论直接修改生产策略逻辑。

---

## 41. 多模型分析与复盘预留

未来系统可能接入 DeepSeek、OpenAI、Grok 或其他多个大模型。

多个大模型不应只接收一个简单的多空结果，而应接收同一份结构化策略分析上下文，并分别输出结构化分析结果。

系统后续应预留对大模型分析结果进行复盘的能力。

复盘对象不应只包括代码策略本身，也应包括：

1. 单个大模型对策略信号的分析质量。
2. 多个大模型之间的一致性和分歧。
3. 大模型是否正确识别策略冲突。
4. 大模型是否正确建议 `wait` 或 `stop_trading`。
5. 大模型是否放大了错误策略信号。
6. 大模型是否过滤掉了低质量交易。
7. 策略组合与大模型组合的配合效果。
8. 大模型建议候选与最终建议之间的差异。
9. 最终建议结果与实际市场表现之间的关系。

例如未来可以评估：

```text
GannStrategy + TrendStrategy + DeepSeek
GannStrategy + TrendStrategy + OpenAI
GannStrategy + TrendStrategy + Grok
GannStrategy + TrendStrategy + MultiModelConsensus
```

但该能力复杂度较高，不属于第一阶段和早期策略阶段的首要开发内容。

当前阶段只要求保留必要的数据链路和模块边界，避免未来无法追溯大模型输入、输出和最终建议来源。

完整评估规则应由 `07_strategy_evaluation_requirements.md` 定义。

---

## 42. 未来复盘对象分层

未来复盘不应只看最终建议结果。

系统后续应能够区分以下复盘对象：

```text
strategy_signal_result
aggregation_result
llm_analysis_result
model_consensus_result
strategy_advice_result
manual_execution_result
```

含义：

1. `strategy_signal_result`：单个代码策略本身是否有效。
2. `aggregation_result`：策略聚合层是否正确识别冲突和风险。
3. `llm_analysis_result`：单个大模型分析是否有效。
4. `model_consensus_result`：多个模型之间的共识或分歧是否有价值。
5. `strategy_advice_result`：最终建议是否有效。
6. `manual_execution_result`：用户实际执行是否偏离建议，以及偏离后结果如何。

该能力不要求早期实现。

但未来设计数据库、日志和模块边界时，应避免丢失这些复盘所需的数据链路。

---

## 43. 第一阶段和早期策略阶段不应过度复杂化

多策略、多模型、复盘评估体系最终可能很复杂。

但早期不应一次性实现完整复杂系统。

正确优先级：

```text
第一阶段：行情数据采集和提醒
第二阶段：单策略或少量策略信号
第三阶段：策略聚合和基础建议
第四阶段：大模型综合分析
第五阶段：建议生命周期和复盘
第六阶段：多模型比较和策略-模型组合评估
```

当前文档只要求预留方向，不要求第一阶段或早期策略阶段全部开发。

---

## 44. Codex 开发约束

Codex 在未来实现策略层时，必须遵守：

1. 先阅读本文档。
2. 先阅读 `03_database_and_quality_requirements.md`。
3. 先阅读 `04_alerting_requirements.md`。
4. 不得实现自动下单。
5. 不得实现自动平仓。
6. 不得实现自动调仓。
7. 不得实现账户同步后自动操作。
8. 策略必须一个策略一个独立模块。
9. 不得把江恩、趋势、支撑压力、风控混在一个大函数里。
10. 每个策略必须输出结构化 `strategy_signal`。
11. 多策略必须先独立运行，再统一聚合。
12. 每个策略信号必须可单独保存。
13. 策略聚合层不得直接跳过大模型分析和风控硬校验生成最终建议。
14. 大模型只能输出 `llm_advice_candidate`，不能直接生成最终 `strategy_advice`。
15. 最终建议必须经过代码风控硬校验。
16. 不得让每个策略各自直接给用户发送最终建议。
17. 最终建议必须包含策略简要结论、理由、分歧分析、大模型综合分析和综合建议。
18. 多策略严重分歧时，必须允许输出 `stop_trading`。
19. 停止交易是有效策略，不是系统失败。
20. 如果风控策略导致 `wait` 或 `stop_trading`，必须明确告诉用户原因。
21. 4h 是主策略周期。
22. 1m 不得单独推翻 4h 主策略。
23. 10s 价格监控不得独立生成策略建议。
24. 基础系统提醒不得调用大模型。
25. 大模型不得绕过代码策略直接生成最终建议。
26. 大模型不得只接收简单多空结果。
27. 大模型输入必须优先结构化。
28. 大模型输出必须尽量结构化。
29. 大模型输入输出必须保存，便于后续复盘。
30. 多模型协作默认采用独立分析和结构化比较，不实现模型自由聊天。
31. 策略运行必须检查数据质量。
32. 策略版本必须记录。
33. 策略参数版本必须记录。
34. 每个策略应优先支持独立配置文件。
35. 策略参数不得硬编码在业务代码中。
36. 策略参数变化必须升级 `parameter_version`。
37. 低频策略不得默认生成过窄开仓区间和过小目标空间。
38. 如果预期利润空间不足以覆盖手续费、滑点和风险成本，应输出 `wait`。
39. 策略输入范围必须可追溯。
40. 大模型不得直接修改并上线核心策略代码。
41. 多模型复盘和策略-模型组合评估属于后期可选扩展，不得在第一阶段提前开发。
42. 不得提交 `.env`、密钥、日志文件或本地缓存文件。
