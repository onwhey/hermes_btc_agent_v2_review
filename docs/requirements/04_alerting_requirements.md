# 04_alerting_requirements.md

# 提醒与告警需求

## 1. 文档目的

本文档定义 Hermes + DeepSeek BTC 合约策略辅助系统中的提醒与告警需求。

本文档重点约束：

1. 哪些事件必须提醒。
2. 哪些事件不能重复刷屏。
3. 提醒等级如何划分。
4. 提醒如何落库。
5. 提醒如何通过 Hermes 发送。
6. Hermes 返回结果如何保存。
7. 发送失败如何处理。
8. 基础系统通知与大模型通知的边界。
9. 后续策略建议提醒如何预留。

本文档不负责：

1. 具体 Python 代码实现。
2. Hermes 官方配置文件完整示例。
3. DeepSeek 策略分析逻辑。
4. Admin 后台展示细节。
5. 自动交易相关任何功能。

本文档只定义 Hermes 作为提醒出口的业务需求和约束，不提供完整 Hermes Gateway 配置教程。

具体 Hermes 配置、route 名称、secret 配置、payload 格式和部署命令，应在对应开发计划或部署文档中按照官方文档实现。

具体实现步骤由 `docs/plans/` 下的阶段计划文件定义。

---

## 2. 提醒系统定位

提醒系统是本项目的核心基础能力之一。

系统提醒的目的不是制造更多噪音，而是帮助用户在关键事件发生时及时知道：

1. 行情数据是否异常。
2. K线采集是否失败。
3. 数据质量是否出现问题。
4. 实时价格是否触发关键事件。
5. 后续策略建议是否需要用户关注。
6. 后续风险条件是否触发。
7. 后续复盘结果是否需要查看。

提醒系统必须遵循：

```text
有事提醒
没事不提醒
同一件事不重复刷屏
重要异常不能静默
```

---

## 3. 当前阶段范围

第一阶段提醒系统只覆盖数据采集层和实时价格事件提醒。

第一阶段必须支持：

1. K线采集异常提醒。
2. K线数据质量异常提醒。
3. K线缺口提醒。
4. K线写库失败提醒。
5. Binance REST 请求失败提醒。
6. Binance WebSocket 最新价格异常提醒。
7. 10s 实时价格事件提醒。
8. MySQL 连接异常提醒。
9. Redis 连接异常提醒。
10. Hermes 发送失败记录。
11. `alert_message` 提醒记录落库。
12. `channel_response` 保存 Hermes 返回结果。
13. 提醒去重和冷却。
14. UTC 和 PRC 时间字段记录。
15. 基础系统通知固定模板生成。

第一阶段不实现：

1. DeepSeek 策略建议提醒。
2. 多策略分析提醒。
3. 策略复盘提醒。
4. Admin 后台提醒管理。
5. 用户分组通知。
6. 多通知渠道轮询。
7. 自动交易相关提醒。
8. 账户、订单、持仓相关提醒。
9. 使用大模型生成基础系统告警正文。

但第一阶段提醒系统的设计不得阻碍后续策略建议、复盘、风控提醒扩展。

---

## 4. Hermes 是统一提醒出口

本项目统一通过 Hermes 发送用户提醒。

第一阶段正式提醒出口为：

```text
Hermes → 微信
```

系统内部不应绕过 Hermes 直接对接其他通知方式。

禁止第一阶段直接实现：

1. 企业微信机器人直连。
2. Telegram 机器人直连。
3. 邮件通知。
4. 短信通知。
5. 自定义第三方 webhook 通知。
6. 浏览器推送。
7. 手机 App 推送。

如果后续确实要增加其他通知方式，也必须通过新的设计决策文档和独立计划文件完成。

## 5. Hermes 开发资料来源约束

凡涉及 Hermes Gateway、Webhook、route、微信投递、payload 格式、认证、签名、返回值处理的开发，必须优先参考 Hermes 官方文档。

官方文档入口由项目文档或配置说明中明确指定。

要求：

1. Codex 实现 Hermes 相关代码时，必须以 Hermes 官方文档为准。
2. 不得依据第三方博客、非官方示例、旧项目残留代码直接实现 Hermes 相关功能。
3. 如果官方文档与旧代码不一致，以官方文档为准。
4. 如果实现时发现官方文档不清楚，必须先停止并让用户确认，不得自行猜测接口格式。
5. Hermes 相关认证、签名、payload 字段必须可测试、可审计、可回滚。

该约束也应写入：

```text
AGENTS.md
docs/decisions/0004-alerting-through-hermes.md
```

## 6. 大模型调用边界

提醒系统必须区分：

1. 系统事件提醒。
2. 行情数据质量提醒。
3. 实时价格事件提醒。
4. 策略建议提醒。
5. 策略复盘提醒。

除操作建议、策略分析、策略冲突解释、策略复盘总结等确实需要大模型思考和解释的通知外，其他通知一律由代码直接生成固定模板，并直接调用 Hermes 发送。

以下提醒禁止调用 DeepSeek、OpenAI、Grok 或其他大模型：

1. K线采集失败。
2. K线不连续。
3. K线缺口。
4. 数据质量检查失败。
5. Binance REST 请求失败。
6. Binance WebSocket 异常。
7. MySQL 异常。
8. Redis 异常。
9. Hermes 发送失败。
10. 实时价格波动提醒。
11. 价格触发基础阈值提醒。
12. 系统运行异常。
13. 配置缺失异常。
14. 日志轮转异常。
15. 任务调度异常。

原因：

1. 系统告警必须稳定、直接、可审计。
2. 基础异常不需要大模型推理。
3. 大模型可能产生不稳定表达。
4. 大模型调用可能失败、变慢或增加成本。
5. 基础告警不能被模型改写成交易建议。
6. 基础告警必须能在 DeepSeek 或其他模型不可用时正常发送。

允许调用大模型的通知仅限后续策略阶段，例如：

1. 4h 主策略建议。
2. 多策略结果解释。
3. 策略冲突分析。
4. 停止交易原因解释。
5. 策略建议复盘。
6. 周期性策略表现总结。

即使后续策略通知调用大模型，最终发送仍必须通过 Hermes，且必须保存模型输入、模型输出、最终提醒内容和 `alert_message` 记录。

## 7. 提醒模板要求

第一阶段提醒内容必须由代码中的固定模板生成。

第一阶段禁止使用 DeepSeek 或其他大模型生成提醒正文。

原因：

1. 第一阶段目标是数据采集和提醒基础层。
2. K线异常提醒必须稳定、清晰、可审计。
3. 大模型生成内容可能不稳定。
4. 数据异常提醒不能被模型改写成交易建议。
5. 基础系统告警不能依赖大模型可用性。

后续策略建议阶段如需使用大模型生成解释文本，必须在策略建议相关文档中单独定义。

---

## 8. 提醒事件分类

系统提醒事件应按业务类型分类。

第一阶段至少支持以下提醒类型：

```text
kline_collection_failed
kline_data_quality_failed
kline_gap_detected
kline_write_failed
binance_rest_failed
binance_ws_failed
price_volatility
price_near_key_level
mysql_unavailable
redis_unavailable
hermes_send_failed
system_runtime_error
config_missing
task_schedule_failed
```

后续阶段可扩展：

```text
strategy_advice_created
strategy_advice_updated
strategy_advice_invalidated
strategy_advice_completed
strategy_advice_expired
strategy_review_ready
strategy_conflict_detected
stop_trading_recommended
manual_action_required
```

提醒类型必须稳定，不应随意改名。

提醒类型用于：

1. 落库。
2. 去重。
3. 冷却。
4. 查询。
5. Admin 后台筛选。
6. 后续复盘关联。

## 9. 提醒等级

系统提醒至少分为以下等级：

```text
info
warning
critical
```

### 9.1 info

`info` 用于普通信息提醒。

例如：

1. 采集任务恢复正常。
2. 后续策略建议生成成功。
3. 后续复盘报告生成完成。

第一阶段应谨慎使用 `info`，避免无意义消息过多。

---

### 9.2 warning

`warning` 用于需要用户关注，但未必立即影响系统核心数据可信度的事件。

例如：

1. 实时价格波动超过普通阈值。
2. 价格接近后续有效建议观察区。
3. Binance WebSocket 短暂断开但已恢复。
4. Redis 短暂异常但不影响核心 MySQL 数据。
5. Hermes 单次发送失败但重试成功。

---

### 9.3 critical

`critical` 用于高优先级异常。

以下事件必须视为 `critical`：

1. 4h K线采集失败。
2. 4h K线解析失败。
3. 4h K线未收盘却准备入库。
4. 4h K线本批次不连续。
5. 4h K线与数据库已有数据不连续。
6. 发现 4h K线历史缺口。
7. 4h K线 OHLC 数据不合法。
8. 4h K线时间字段异常。
9. 4h K线写入 MySQL 失败。
10. 数据质量检查失败并阻断写库。
11. MySQL 不可用。
12. 连续多次 Binance REST 请求失败。
13. Hermes 连续发送失败。

K线数据是后续策略判断、建议生成和复盘验证的核心数据源。

因此，只要 K线采集链路出现关键异常，必须发送 `critical` 级别提醒。

---

## 10. K线采集异常强制提醒规则

只要 K线采集、解析、校验、连续性检查、回补、写库任一关键步骤失败，系统必须：

1. 阻止脏数据进入行情主表。
2. 写入 `collector_event_log`。
3. 写入 `data_quality_check`。
4. 写入 `alert_message`。
5. 通过 Hermes 发送微信提醒。

以下情况必须提醒：

1. Binance REST 请求失败。
2. Binance REST 返回数据为空。
3. Binance REST 返回格式异常。
4. K线解析失败。
5. 返回 K线未收盘。
6. 本批次 K线不连续。
7. 本批次与数据库已有 K线不连续。
8. 发现历史 K线缺口。
9. OHLC 数据不合法。
10. 数值字段异常。
11. 时间字段异常。
12. 数据来源异常。
13. 同一批次存在重复业务唯一键。
14. K线写入 MySQL 失败。
15. 数据质量检查失败。
16. 自动回补失败。
17. 连续多次采集任务失败。

要求：

1. K线异常不能静默失败。
2. K线异常不能只写日志文件。
3. K线异常必须落库。
4. K线异常必须通过 Hermes 提醒。
5. 同一异常事件可以做去重和冷却。
6. 去重和冷却不能导致异常完全不提醒。
7. K线异常默认是高级别告警。
8. K线异常提醒必须由代码模板生成，不得调用大模型。

---

## 11. 实时价格提醒规则

系统可以每 10 秒检查一次 BTCUSDT 最新价格。

但 10 秒是检查频率，不是提醒频率。

实时价格监控必须遵循：

```text
有事提醒
没事不提醒
同一事件不重复提醒
```

禁止：

1. 每 10 秒发送一次价格。
2. 每 10 秒调用 DeepSeek。
3. 每 10 秒调用任何大模型。
4. 每 10 秒生成策略建议。
5. 每 10 秒推翻 4h 主策略。
6. 对同一价格事件重复刷屏。

实时价格提醒可以用于：

1. 短时间价格波动超过阈值。
2. 价格接近关键支撑。
3. 价格接近关键压力。
4. 价格突破关键位。
5. 价格跌破关键位。
6. 价格接近后续有效建议的观察区。
7. 价格进入后续计划开仓区。
8. 价格接近后续止损失效区。
9. 价格接近后续止盈或减仓观察区。
10. 价格出现异常插针或剧烈波动。

第一阶段如果尚未实现策略建议系统，则实时价格提醒只能基于基础阈值或手动配置关键位，不得假装已经具备完整策略判断能力。

实时价格提醒必须由代码模板生成，不得调用大模型。

---

## 12. 实时价格提醒阈值配置

实时价格提醒相关阈值必须来自配置文件或数据库配置，不得硬编码在业务代码中。

至少应支持配置：

1. 价格波动百分比阈值。
2. 价格波动检查窗口。
3. 提醒冷却时间。
4. 关键价位接近距离。
5. 是否启用实时价格提醒。
6. 是否启用某类价格事件提醒。

第一阶段可以先通过 `.env` 或 YAML 配置管理，后续 Admin 阶段再支持后台配置。

---

## 13. 提醒去重规则

提醒系统必须支持去重。

去重目标：

1. 防止同一事件重复发送。
2. 防止 10s 价格监控刷屏。
3. 防止持续性异常不断轰炸微信。
4. 保证重要异常至少提醒一次。

提醒事件应生成稳定的 `dedupe_key`。

`dedupe_key` 可以由以下字段组成：

```text
alert_type
level
exchange
market_type
symbol
interval
business_key
time_bucket
```

示例：

```text
kline_gap_detected:critical:binance:um_futures:BTCUSDT:4h:20260506T120000Z
```

要求：

1. 同一 `dedupe_key` 在冷却窗口内不得重复发送。
2. 不同异常事件不得被错误合并。
3. K线异常即使被冷却，也必须保留数据库记录。
4. 冷却只能限制微信发送频率，不能阻止异常落库。
5. 关键异常持续存在时，应允许升级提醒。

---

## 14. K线异常 dedupe_key 要求

K线相关异常的 `dedupe_key` 必须包含足够精确的业务定位信息。

例如 K线缺口提醒应包含：

1. alert_type。
2. level。
3. exchange。
4. market_type。
5. symbol。
6. interval。
7. missing_open_time_utc 或 missing_range。
8. check_task_name。

示例：

```text
kline_gap_detected:critical:binance:um_futures:BTCUSDT:4h:20260506T120000Z
```

禁止只使用：

```text
alert_type + symbol + interval
```

作为 K线异常去重 key，否则不同时间的缺口会被错误合并。

---

## 15. 提醒冷却规则

提醒系统必须支持冷却窗口。

冷却窗口用于控制同一类事件的重复提醒频率。

冷却判断必须基于 UTC 时间。

禁止使用 PRC 时间进行冷却判断。

建议规则：

1. 实时价格波动提醒可设置较短冷却。
2. K线采集异常必须首次立即提醒。
3. K线采集异常持续存在时，可以按较长冷却合并提醒。
4. 连续失败达到阈值时，应发送升级提醒。
5. Hermes 发送失败本身也需要记录，但避免无限递归提醒。

冷却只影响发送频率，不影响：

1. 采集事件记录。
2. 数据质量记录。
3. `alert_message` 记录。
4. 错误日志记录。

---

## 16. critical 首次提醒规则

`critical` 级别事件首次发生时，不得被冷却规则直接抑制。

系统必须至少完成：

1. 创建 `alert_message`。
2. 尝试通过 Hermes 发送。
3. 记录 Hermes 返回结果。

同一 `critical` 事件持续存在时，可以进入冷却或合并提醒。

但首次出现必须提醒。

---

## 17. 持续异常升级规则

对于持续性异常，系统不能只提醒一次后永远沉默。

以下情况应支持升级提醒：

1. 同一 K线采集任务连续失败。
2. 同一数据质量异常持续多个周期。
3. 同一 K线缺口长时间未修复。
4. MySQL 连续不可用。
5. Redis 连续不可用。
6. Hermes 连续发送失败。
7. Binance REST 连续请求失败。

升级提醒应包含：

1. 首次发生时间。
2. 最近发生时间。
3. 连续失败次数。
4. 影响交易对。
5. 影响周期。
6. 当前处理状态。
7. 是否已阻断行情写入。
8. 是否需要人工处理。

---

## 18. 恢复提醒规则

对于持续性 `critical` 异常，系统应支持恢复提醒。

例如：

1. K线采集连续失败后恢复成功。
2. MySQL 不可用后恢复。
3. Redis 不可用后恢复。
4. Hermes 连续失败后恢复。
5. Binance REST 连续失败后恢复。

恢复提醒等级可以为 `info` 或 `warning`。

恢复提醒必须去重，不能每次成功都发送。

第一阶段可以只实现核心链路恢复提醒，其他恢复提醒后续再扩展。

---

## 19. 提醒落库规则

所有提醒发送前后都必须写入 `alert_message` 或等价表。

`alert_message` 是提醒记录表，不是策略建议主表。

提醒记录应至少保存：

1. 提醒类型。
2. 提醒等级。
3. 提醒标题。
4. 提醒内容。
5. 交易所。
6. 市场类型。
7. 交易对。
8. K线周期。
9. 关联业务类型。
10. 关联业务 id。
11. 去重 key。
12. 冷却窗口。
13. 发送通道。
14. 发送状态。
15. 发送时间 UTC。
16. 发送时间 PRC。
17. Hermes 返回结果。
18. 失败原因。
19. 重试次数。
20. 创建时间 UTC。
21. 创建时间 PRC。
22. 更新时间 UTC。
23. 更新时间 PRC。

提醒排序、冷却、去重必须使用 UTC 时间。

PRC 时间只用于用户阅读和人工排查。

---

## 20. 发送尝试记录边界

第一阶段不强制建立独立的 `alert_delivery_log` 表。

第一阶段可以在 `alert_message` 中保存：

1. 当前发送状态。
2. 最近一次 Hermes 返回结果。
3. retry_count。
4. failure_reason。
5. sent_at_utc。
6. sent_at_prc。

如果后续需要完整保存每一次发送尝试，再单独设计 `alert_delivery_log`。

Codex 不得在第一阶段提前实现复杂多通道投递日志表。

## 21. alert_message 与 strategy_advice 的关系

`alert_message` 不能替代 `strategy_advice`。

第一阶段不实现完整 `strategy_advice`。

后续当系统进入策略建议阶段时：

1. 策略建议必须保存到独立的 `strategy_advice` 表。
2. 策略信号必须保存到独立的 `strategy_signal` 表。
3. 策略建议提醒通过 `alert_message` 记录发送过程。
4. `alert_message` 应通过 `related_type` 和 `related_id` 关联对应业务对象。
5. 某条提醒如果属于策略建议提醒，应关联对应 `strategy_advice`。
6. 某条提醒如果属于策略复盘提醒，应关联对应复盘结果或建议链。

第一阶段可以预留：

```text
related_type
related_id
```

但不实现完整策略建议表。

## 22. Hermes 发送结果记录

系统必须保存 Hermes 返回结果。

建议字段：

```text
channel_response
```

`channel_response` 应使用 JSON 类型或等价 JSON 文本字段。

不得只保存：

```text
success
failed
```

应尽量保存：

1. Hermes 返回状态。
2. Hermes 返回消息摘要。
3. HTTP 状态码。
4. 错误类型。
5. 错误摘要。
6. 响应耗时。
7. route 名称。
8. 发送目标通道。
9. 请求时间 UTC。
10. 响应时间 UTC。
11. 是否重试。
12. 重试次数。

要求：

1. `channel_response` 不得保存密钥。
2. `channel_response` 不得保存完整敏感配置。
3. Hermes Webhook secret 不得入库。
4. 如果 Hermes 返回内容过大，只保存摘要和关键字段。
5. Hermes 发送失败时，也必须更新 `alert_message` 状态。

## 23. Hermes 发送失败处理

Hermes 发送失败不能导致原始异常丢失。

例如 K线采集失败时，即使 Hermes 也发送失败，系统仍必须保存：

1. `collector_event_log`。
2. `data_quality_check`。
3. `alert_message`。
4. 本地错误日志。

Hermes 发送失败时：

1. `alert_message.status` 应标记为失败。
2. `failure_reason` 应保存失败摘要。
3. `channel_response` 应保存返回摘要。
4. 可按配置进行有限次数重试。
5. 重试次数必须记录。
6. 不能无限重试。
7. 不能因为提醒发送失败导致主流程死循环。

如果 Hermes 连续发送失败，应记录系统级异常。

但要注意：Hermes 本身失败时，不能通过 Hermes 无限提醒 Hermes 失败，必须避免递归告警。

## 24. Hermes 失败兜底边界

第一阶段 Hermes 发送失败时，系统必须记录失败，但不得自动切换到其他第三方通知渠道。

允许的兜底方式：

1. 写入 `alert_message`。
2. 写入本地日志。
3. 写入 `collector_event_log` 或系统错误日志。
4. 在后续健康检查中显示 Hermes 异常状态。

禁止第一阶段自动改用：

1. 邮件。
2. Telegram。
3. 企业微信机器人直连。
4. 短信。
5. 其他第三方 webhook。

其他通知渠道必须通过新的设计决策和计划文件引入。

## 27. 实时价格提醒与策略建议的边界

实时价格提醒不是策略建议。

2. 立即做空。
3. 立即加仓。
4. 立即平仓。
5. 自动下单。
6. 自动调仓。
7. DeepSeek 已确认机会。

后续如果实时价格触发当前有效策略建议的观察区，也必须基于已存在的 `strategy_advice`，不能让 10s 价格监控独立生成新策略。

## 30. 提醒状态

`alert_message` 的发送状态至少包括：

```text
pending
sent
failed
skipped
suppressed
retrying
```

说明：

1. `pending`：已创建，尚未发送。
2. `sent`：发送成功。
3. `failed`：发送失败。
4. `skipped`：按规则跳过发送。
5. `suppressed`：因去重或冷却被抑制发送。
6. `retrying`：正在重试或等待重试。

注意：

1. 被冷却抑制的提醒仍可记录为 `suppressed`。
2. `suppressed` 不等于事件不存在。
3. critical K线异常首次出现时不得直接 suppressed。
4. 同一异常持续存在时可以 suppressed，但必须保留事件记录。

## 31. 提醒事务边界

提醒系统必须有清晰的事务边界。

要求：

1. 业务异常记录不能因为 Hermes 发送失败而丢失。
2. `alert_message` 创建失败不能导致行情主表写入错误数据。
3. K线质量失败时，行情主表必须不写入或回滚。
4. K线质量失败时，异常记录和提醒记录必须尽量保留。
5. Hermes 发送应在核心异常记录保存后执行。
6. 如果发送失败，应更新提醒状态，而不是回滚全部异常记录。
