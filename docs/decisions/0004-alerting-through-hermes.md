# 0004-alerting-through-hermes.md

# 决策 0004：系统提醒统一通过 Hermes 发送

## 1. 决策状态

已确定。

本决策用于约束系统中的提醒发送方式、提醒边界、Hermes 集成方式、基础告警是否调用大模型、提醒记录入库、去重冷却和后续策略建议提醒的关联规则。

后续数据采集、数据质量检查、价格监控、策略建议、复盘提醒、Admin 展示和代码实现都必须遵守本决策。

---

## 2. 决策背景

本项目需要把重要事件及时通知用户。

当前主要提醒场景包括：

1. K线采集失败。
2. K线数据不连续。
3. 数据质量异常。
4. Binance REST 请求异常。
5. MySQL 异常。
6. Redis 异常。
7. Hermes 发送异常。
8. 10s 价格波动提醒。
9. 后续价格接近关键位提醒。
10. 后续策略建议提醒。
11. 后续建议复盘提醒。

用户已经使用 Hermes 连接个人微信。

因此，系统提醒不应绕过 Hermes 直接对接微信，也不应各业务模块各自实现一套发送逻辑。

系统需要统一提醒通道，保证：

1. 所有提醒可追溯。
2. 所有提醒有记录。
3. 所有提醒可去重和冷却。
4. 所有提醒可以区分类型和级别。
5. 所有提醒通过统一 Hermes 通道发送。
6. 基础系统提醒不调用大模型。
7. 策略建议、复盘等需要思考的内容，后续可以进入大模型分析流程。

---

## 3. 决策内容

本项目确定以下规则：

1. 所有微信提醒统一通过 Hermes 发送。
2. 业务代码不得直接调用微信接口。
3. 业务代码不得绕过统一 alerting 模块发送提醒。
4. 当前推荐通过 Hermes Webhook Adapter 发送提醒。
5. 系统基础提醒使用代码模板直接生成。
6. 基础提醒不得调用 DeepSeek、OpenAI、Grok 或其他大模型。
7. 策略建议、策略分析、复盘总结等确实需要推理和解释的内容，未来可以调用大模型。
8. 每条提醒都必须写入 `alert_message` 或等价提醒记录表。
9. Hermes 返回结果必须记录到 `channel_response` 或等价字段。
10. 提醒消息中如出现时间，必须同时标注北京时间和 UTC。
11. 提醒系统必须支持去重和冷却，防止重复刷屏。
12. 后续策略建议提醒应能关联 `strategy_advice` 表。
13. 系统不得因为提醒触发而自动交易。

---

## 4. Hermes 作为统一提醒通道

Hermes 是本项目对外提醒的统一出口。

系统内部各业务模块不应直接关心微信如何发送。

推荐数据流：

1. 业务模块发现事件。
2. 构造结构化 `AlertEvent`。
3. 调用统一 `AlertDispatcher`。
4. `AlertDispatcher` 写入 `alert_message`。
5. `HermesWebhookChannel` 调用 Hermes Webhook。
6. Hermes 通过微信通知用户。
7. 系统记录 Hermes 返回结果。

业务模块只负责告诉 alerting 模块：

1. 发生了什么。
2. 事件类型是什么。
3. 严重级别是什么。
4. 涉及哪个交易对。
5. 涉及哪个周期。
6. 需要展示什么内容。
7. 是否需要冷却。
8. 是否需要关联业务对象。

业务模块不负责：

1. 直接拼微信接口。
2. 直接调用微信发送。
3. 直接读取 Hermes secret。
4. 直接处理 Hermes HTTP 请求细节。
5. 各自实现提醒重试和冷却。

---

## 5. Hermes Webhook Adapter 规则

当前项目提醒通道优先采用 Hermes Webhook Adapter。

规则：

1. Hermes Webhook 地址、route、secret 等配置必须通过统一配置模块读取。
2. 不得在业务代码中硬编码 Hermes secret。
3. 不得把 Hermes secret 提交到 Git。
4. 不得把 Hermes secret 写入日志。
5. 请求 Hermes 时应携带必要认证信息。
6. Hermes 返回结果应记录到 `alert_message.channel_response`。
7. Hermes 请求失败应记录失败状态。
8. Hermes 请求失败不应导致已经确认可靠的 K线数据回滚。
9. Hermes 请求失败本身也应成为可排查事件。
10. Hermes 集成应参考 Hermes 官方文档，不得依赖第三方不可靠资料。

Hermes 官方文档：

1. `https://hermes-agent.nousresearch.com/docs/`

注意：

1. 本项目不直接依赖第三方微信机器人。
2. 本项目不直接调用个人微信接口。
3. 本项目不绕过 Hermes 建立其他微信发送通道。
4. 后续如 Hermes 官方接口变化，应以 Hermes 官方文档为准调整适配层。

---

## 6. Direct Delivery 规则

基础系统提醒不需要 Hermes 调用大模型进行推理。

Hermes route 应优先使用 direct delivery 模式。

推荐原则：

1. 基础提醒由代码生成固定模板。
2. Hermes 只负责把内容发送到微信。
3. 基础提醒不得让 Hermes 再调用模型思考。
4. 基础提醒不得因为模型不可用而无法发送。

适合 direct delivery 的提醒包括：

1. K线采集失败。
2. K线数据不连续。
3. 数据质量异常。
4. Binance 接口异常。
5. MySQL 连接异常。
6. Redis 连接异常。
7. 程序运行异常。
8. 价格波动提醒。
9. Hermes 通道发送失败记录。

不适合 direct delivery、未来可能需要大模型参与的内容包括：

1. 策略综合分析。
2. 多策略分歧解释。
3. 操作建议生成。
4. 复盘总结。
5. 不同大模型候选建议对比。
6. 策略长期表现分析。

---

## 7. 基础提醒不得调用大模型

除操作建议、策略分析、复盘总结等确实需要大模型思考和解释的通知外，其他系统通知一律由代码直接调用 Hermes 发送。

以下提醒不得调用大模型：

1. K线采集异常。
2. K线缺口异常。
3. K线不连续异常。
4. 数据质量异常。
5. Binance REST 请求异常。
6. Binance 数据解析异常。
7. MySQL 异常。
8. Redis 异常。
9. Hermes 通道异常。
10. 价格波动超过阈值。
11. 任务启动、任务完成、任务失败等基础运行状态。

原因：

1. 基础系统异常必须快速、确定、可控。
2. 基础告警不需要大模型解释。
3. 大模型可能产生幻觉。
4. 大模型不可用不应阻塞基础告警。
5. 基础告警内容应标准化，方便排查和复盘。

---

## 8. alert_message 表职责

`alert_message` 是提醒记录表。

它的主要职责是记录系统已经产生和发送过的提醒。

`alert_message` 应记录：

1. 提醒标题。
2. 提醒内容。
3. 提醒类型。
4. 提醒级别。
5. 交易对。
6. 周期。
7. 业务对象类型。
8. 业务对象 id。
9. 发送渠道。
10. 发送状态。
11. Hermes 返回结果。
12. 错误信息。
13. UTC 创建时间。
14. PRC 创建时间。
15. 是否已发送。
16. 是否被冷却或去重。
17. 重试次数。

说明：

1. `alert_message` 不是策略建议主表。
2. `alert_message` 可以记录类型为 `strategy_advice` 的提醒。
3. 但后续策略建议核心数据必须保存在专门的 `strategy_advice` 表中。
4. `alert_message` 中的策略建议提醒应通过外键或业务引用关联 `strategy_advice`。
5. `alert_message` 用于提醒追踪，不用于替代建议生命周期。

---

## 9. strategy_advice 关联规则

后续进入策略建议阶段后，系统应建立专门的 `strategy_advice` 表。

原因：

1. 本项目核心是策略建议和长期复盘。
2. 策略建议需要生命周期。
3. 策略建议需要版本链。
4. 策略建议需要关联策略信号。
5. 策略建议需要关联大模型输入输出。
6. 策略建议需要关联复盘结果。
7. 策略建议不能只保存在提醒消息中。

`alert_message` 与 `strategy_advice` 的关系：

1. `strategy_advice` 保存建议本体。
2. `alert_message` 保存该建议是否通知过用户。
3. 一条策略建议可以有一条或多条提醒记录。
4. 提醒记录应能追溯到对应建议。
5. 提醒失败不应导致建议本体丢失。
6. 建议本体变更不应通过修改提醒消息实现。

## 10. 提醒类型

系统提醒类型应结构化。

推荐提醒类型包括：

1. `collector_failed`
2. `data_quality_failed`
3. `kline_gap_detected`
4. `kline_overlap_mismatch`
5. `binance_rest_error`
6. `mysql_error`
7. `redis_error`
8. `hermes_error`
9. `price_movement`
10. `price_near_key_level`
11. `strategy_advice`
12. `advice_update`
13. `advice_close`
14. `review_due`
15. `review_result`
16. `system_health`

提醒类型不得随意自由拼接。

如需新增提醒类型，应在相关文档和代码枚举中同步说明。

---

## 11. 提醒级别

系统提醒应支持级别。

推荐级别：

1. `info`
2. `warning`
3. `error`
4. `critical`

级别含义：

1. `info`：普通信息提醒，不一定需要立即处理。
2. `warning`：需要关注，但不一定影响核心数据。
3. `error`：已经出现错误，可能影响数据或任务。
4. `critical`：严重错误，可能影响核心数据、策略判断或系统稳定性。

K线采集失败和数据质量异常应至少视为 `error`。

严重 K线缺口、连续性异常、官方数据与数据库重叠数据不一致，应视为高优先级异常，必要时可使用 `critical`。

数据是本项目核心资产，数据采集异常不得被低优先级处理。

---

## 12. 去重与冷却规则

提醒系统必须防止刷屏。

需要去重或冷却的场景包括：

1. 同一个 Binance REST 异常持续发生。
2. 同一个 K线缺口反复被检测到。
3. 同一个 Redis 异常反复出现。
4. 同一个 MySQL 异常反复出现。
5. 价格在短时间内反复超过阈值。
6. Hermes 通道短时间持续失败。
7. 同一条建议复盘提醒重复触发。

规则：

1. 同一异常不得无限刷屏。
2. 冷却状态可以保存在 Redis。
3. 关键提醒即使冷却，也应保留必要的数据库记录或计数。
4. 冷却不能导致严重异常完全不可见。
5. 同一条建议链的复盘提醒只能发送一次。
6. 提醒去重不得影响正式业务数据写入和质量判断。
7. 去重依据应包含提醒类型、交易对、周期、业务对象、异常指纹等信息。

---

## 13. 消息内容规则

提醒消息必须简洁、明确、可行动。

基础提醒消息应包含：

1. 标题。
2. 提醒类型。
3. 提醒级别。
4. 交易对。
5. 周期。
6. 发生时间。
7. 影响范围。
8. 异常原因。
9. 系统处理结果。
10. 是否已停止写库。
11. 是否需要用户关注。
12. 消息 id 或追踪 id。

涉及时间时，必须同时展示：

1. 北京时间。
2. UTC。

禁止只写没有时区的时间，例如：

1. `04:00`
2. `12:00`
3. `2026-05-06 04:00`

推荐格式：

1. 北京时间：2026-05-06 12:00:00
2. UTC：2026-05-06 04:00:00

禁止在面向用户的消息中单独使用 `CST` 表示北京时间。

---

## 14. K线异常提醒规则

K线采集和质量异常是高优先级提醒。
K线一致性复核异常属于基础系统报警，必须由代码使用固定模板直接调用 Hermes 发送，不允许调用 DeepSeek 或其他大模型生成报警内容。

该报警的目的不是自动修复数据，而是提醒用户检查 Binance REST 访问状态、采集代码、调度器、数据库写入逻辑、未收盘 K线过滤逻辑或 K线连续性检查逻辑。

K线一致性复核报警不得触发自动回补、自动覆盖或自动修改正式 K线表。

必须提醒的场景包括：

1. 4h K线采集失败。
2. Binance REST 请求失败。
3. Binance 返回异常数据。
4. K线解析失败。
5. K线未收盘却准备写库。
6. K线时间不连续。
7. 数据库缺少中间 K线。
8. 重叠 K线与数据库已有记录不一致。
9. 数据质量检查失败。
10. MySQL 写入失败。

规则：

1. 只要 K线采集出现问题，必须通过 Hermes 提醒。
2. K线异常不得只写日志。
3. K线异常必须写 `collector_event_log`。
4. 数据质量异常必须写 `data_quality_check`。
5. 数据质量失败时，不得写入正式 K线表。
6. 基础 K线异常提醒不得调用大模型。
7. K线异常提醒必须说明是否已经停止写库。
8. K线异常提醒必须说明是否需要人工排查。


如果 MySQL 不可用：
1. 不得假装已落库。
2. 必须写本地 emergency 日志。
3. 如 Redis 可用，可写短期 outbox / failure key。
4. 如 Hermes 可用，必须直接发送“数据库不可用”提醒。
5. MySQL 恢复后，可以由恢复任务补写故障摘要，但不得伪造原始发生时间。

## 15. 10s 价格提醒规则

10s 价格监控只用于事件提醒。

允许提醒：

1. 价格快速波动。
2. 价格超过设定阈值。
3. 后续价格接近入场区。
4. 后续价格接近止盈区。
5. 后续价格接近止损或失效条件。
6. 后续价格接近减仓区。
7. 后续价格接近成本线或保护利润位置。

禁止：

1. 每 10s 无脑提醒。
2. 价格提醒直接生成正式交易建议。
3. 价格提醒自动下单。
4. 价格提醒自动平仓。
5. 价格提醒自动调仓。
6. 价格提醒绕过 Hermes。
7. 价格提醒调用大模型解释基础波动。

价格提醒必须具备：

1. 阈值。
2. 冷却。
3. 去重。
4. 落库记录。
5. Hermes 发送结果记录。

---

## 16. Hermes 发送失败处理

Hermes 发送失败必须可追踪。

规则：

1. Hermes 发送失败必须记录到 `alert_message`。
2. 失败原因应记录在错误字段或 `channel_response` 中。
3. 发送失败不得静默忽略。
4. 发送失败不得导致基础业务假装成功。
5. 发送失败不得导致已确认可靠的 K线数据回滚。
6. 系统可以根据后续设计增加重试机制。
7. 重试必须有次数限制。
8. 重试不得造成无限请求。
9. Hermes 通道持续失败时，应在日志和数据库中保留足够排查信息。

注意：

1. 如果 K线写库已经成功，但 Hermes 提醒失败，不应回滚 K线数据。
2. 如果 K线质量失败，本来就不应写正式 K线表，Hermes 失败也不得改变质量判断结果。
3. 提醒发送状态和业务处理状态是两个不同概念。

---

## 17. alerting 模块边界

系统应建立统一 alerting 模块。

推荐职责：

1. 接收结构化 `AlertEvent`。
2. 生成提醒消息。
3. 执行去重和冷却判断。
4. 写入 `alert_message`。
5. 调用 Hermes 通道。
6. 记录 Hermes 返回结果。
7. 记录发送失败状态。

alerting 模块不得负责：

1. 采集 Binance 数据。
2. 判断 K线是否连续。
3. 写入正式 K线表。
4. 执行策略分析。
5. 调用大模型生成基础告警。
6. 自动交易。
7. 自动下单。
8. 自动平仓。
9. 自动调仓。

Repository 不得直接调用 Hermes。

Binance client 不得直接调用 Hermes。

Strategy 不得直接调用 Hermes 发送最终建议。

策略建议应进入建议生命周期和统一提醒流程。

---

## 18. 配置规则

Hermes 相关配置必须通过统一配置模块读取。

配置项可以包括：

1. Hermes Webhook URL。
2. Hermes route。
3. Hermes secret。
4. Hermes timeout。
5. Hermes enabled。
6. 默认提醒渠道。
7. 冷却时间。
8. 重试次数。
9. 是否记录 channel response。

规则：

1. 不得在业务代码中硬编码 Hermes secret。
2. 不得提交真实 `.env`。
3. `.env.example` 可以提供示例配置，但不得包含真实密钥。
4. 敏感配置不得写入日志。
5. 生产环境和开发环境配置应隔离。
6. 测试环境应允许 mock Hermes 通道。

---

## 19. 测试与验收规则

Hermes 提醒模块至少应能验证以下路径：

1. 正常发送提醒。
2. Hermes 返回成功。
3. Hermes 返回失败。
4. Hermes 请求超时。
5. 去重生效。
6. 冷却生效。
7. alert_message 正常写入。
8. channel_response 正常记录。
9. 基础提醒不调用大模型。
10. 业务模块通过统一 dispatcher 发送提醒。
11. 不绕过 Hermes 直接调用微信。
12. 时间展示包含北京时间和 UTC。

K线相关模块验收时，应验证：

1. 采集失败会触发 Hermes 提醒。
2. K线不连续会触发 Hermes 提醒。
3. 数据质量失败会触发 Hermes 提醒。
4. 数据质量失败不会写正式 K线表。
5. Hermes 失败不会掩盖原始数据质量问题。

---

## 20. 对 Codex 的实现约束

Codex 实现相关代码时必须遵守：

1. 所有微信提醒必须通过 Hermes。
2. 不得直接调用微信接口。
3. 不得绕过统一 alerting 模块。
4. 不得让基础提醒调用大模型。
5. 不得把 Hermes secret 写死在代码里。
6. 不得把 Hermes secret 写入日志。
7. 不得提交真实 `.env`。
8. 必须记录提醒消息。
9. 必须记录 Hermes 返回结果。
10. 必须支持或预留去重和冷却。
11. K线采集失败必须 Hermes 提醒。
12. 数据质量异常必须 Hermes 提醒。
13. 价格提醒必须有阈值和冷却。
14. 提醒中涉及时间时必须标注北京时间和 UTC。
15. `alert_message` 不得替代未来 `strategy_advice`。
16. 后续策略建议提醒必须能关联 `strategy_advice`。
17. Repository 不得直接调用 Hermes。
18. Binance client 不得直接调用 Hermes。
19. 基础提醒不得自动交易。
20. 任何提醒不得触发自动下单、自动平仓或自动调仓。

---

## 21. 决策后果

本决策带来的结果：

1. 系统提醒出口统一。
2. 业务模块不会各自实现微信发送。
3. Hermes 成为微信提醒的统一通道。
4. 基础告警快速、确定、可控。
5. 基础告警不会被大模型不可用影响。
6. 提醒消息可以追溯。
7. Hermes 返回结果可以追踪。
8. 数据质量异常能够及时通知用户。
9. 10s 价格提醒不会刷屏。
10. 后续策略建议提醒可以与 `strategy_advice` 关联。
11. Admin 后续可以查看提醒历史。
12. 复盘时可以知道哪些建议和异常曾经通知过用户。
