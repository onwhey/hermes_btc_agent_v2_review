# 0003-kline-table-splitting.md

# 决策 0003：K线按周期分表

## 1. 决策状态

已确定。

本决策用于约束系统中不同周期 K线数据的表结构边界、数据用途边界、查询边界和后续策略引用边界。

后续数据采集、数据质量检查、策略分析、复盘评估、Admin 展示和代码实现都必须遵守本决策。

---

## 2. 决策背景

本项目当前主策略周期是 `4h`。

后续系统可能会采集和使用多个周期的 K线数据，例如：

1. `1m`
2. `4h`
3. `1d`

这些 K线虽然都来自 Binance REST K线接口，字段结构也高度相似，但它们在系统中的用途、数据量、质量要求和查询方式不同。

例如：

1. `4h` 是当前主策略周期，是策略建议、建议生命周期、复盘评估的核心数据。
2. `1m` 后续主要用于复盘价格路径、插针、快速波动、局部行情细节。
3. `1d` 后续主要用于大级别市场环境判断，例如牛市、熊市、震荡、趋势背景。

如果所有周期都混在一张通用 `market_kline` 表中，短期看实现简单，但长期会带来问题：

1. 不同周期数据量差异巨大。
2. `1m` 数据会远大于 `4h` 和 `1d`。
3. 查询时必须频繁依赖 `interval` 过滤。
4. 索引压力变大。
5. 数据质量检查逻辑容易混乱。
6. 4h 主策略数据容易被大量 1m 数据淹没。
7. 后续策略引用和复盘追溯不够清晰。
8. 后续归档、清理、分区、优化会更困难。
9. Codex 容易把不同周期的业务语义混在一起。

因此，本项目决定按 K线周期进行表级隔离。

---

## 3. 决策内容

本项目采用 K线按周期分表的设计。

当前阶段优先实现：

1. `market_kline_4h`

后续如有需要，再实现：

1. `market_kline_1m`
2. `market_kline_1d`

原则：

1. `4h` K线使用独立正式表。
2. `1m` K线后续使用独立正式表。
3. `1d` K线后续使用独立正式表。
4. 不同周期 K线不得混在同一张正式业务表中。
5. 不同周期可以复用 REST 请求、解析、字段校验、已收盘过滤、连续性检查等通用能力。
6. 不同周期不得混用 Repository、质量检查结果和业务语义。
7. 不同周期之间的关联必须通过 `symbol + open_time_ms` 或 UTC 时间范围完成。
8. 禁止依赖数据库自增 id 关联不同周期 K线。
9. 禁止依赖数据库自增 id 判断 K线顺序。

---

## 4. 表命名规则

正式 K线表命名应直接体现周期。

推荐命名：

1. `market_kline_1m`
2. `market_kline_4h`
3. `market_kline_1d`

当前阶段只需要优先实现：

1. `market_kline_4h`

不建议继续使用一张通用 `market_kline` 表承载所有周期的正式 K线数据。

如果项目早期临时存在通用 `market_kline` 表，该表不得成为后续多周期 K线的长期正式承载表。

后续正式开发应逐步以周期专用表为准。

---

## 5. 各周期表的职责

### 5.1 `market_kline_4h`

`market_kline_4h` 是当前阶段最重要的正式行情表。

它负责保存 Binance REST 返回的官方已收盘 4h K线。

主要用途：

1. 主策略分析。
2. 多策略信号输入。
3. 策略聚合输入。
4. 大模型行情上下文。
5. 操作建议生成依据。
6. 建议生命周期判断。
7. 复盘评估。
8. 数据质量检查。
9. Admin 展示。
10. 长期策略验证。

要求：

1. 数据必须来自 Binance REST 官方已收盘 K线。
2. 不得使用 WebSocket 拼接 4h K线。
3. 未收盘 K线不得写入。
4. 数据连续性必须严格检查。
5. 数据异常必须通过 Hermes 提醒。
6. 业务排序必须基于 UTC 时间字段。
7. 不得依赖自增 id 判断顺序。

---

### 5.2 `market_kline_1m`

`market_kline_1m` 当前阶段可以暂不实现。

后续如果实现，主要用于复盘和细节行情分析。

主要用途：

1. 观察价格路径。
2. 分析插针。
3. 分析快速波动。
4. 分析建议触发后的短周期行情变化。
5. 辅助判断止盈后是否继续沿原方向运行。
6. 辅助判断止损后是否是假突破。
7. 辅助复盘用户实际执行点位附近的市场细节。

`1m` 不应替代 `4h` 主策略周期。

`1m` 不应直接推翻 4h 主策略结构。

`1m` 可以作为辅助证据，但不能成为当前阶段主策略建议的数据核心。

---

### 5.3 `market_kline_1d`

`market_kline_1d` 当前阶段可以暂不实现。

后续如果实现，主要用于大级别市场环境判断。

主要用途：

1. 判断大级别趋势背景。
2. 判断牛市、熊市、震荡市。
3. 辅助判断趋势强弱。
4. 辅助策略环境分类。
5. 辅助评估某个策略在不同市场环境下的表现。
6. 给 4h 策略提供更高周期背景。

`1d` 不应和 `4h` 混在同一张正式业务表中。

---

## 6. 字段设计原则

不同周期 K线表的核心字段应尽量保持一致，方便复用解析、校验、查询和后续分析能力。

每张 K线表建议至少包含以下字段类别：

### 6.1 基础身份字段

1. `id`
2. `exchange`
3. `market_type`
4. `symbol`
5. `interval`
6. `data_source`

说明：

1. 即使表名已经包含周期，也建议保留 `interval` 字段。
2. `interval` 可以用于数据自检、通用逻辑复用和防止写错表。
3. `data_source` 用于标识数据来源，例如 `binance_rest`。

---

### 6.2 时间字段

建议包含：

1. `open_time_ms`
2. `open_time_utc`
3. `open_time_prc`
4. `close_time_ms`
5. `close_time_utc`
6. `close_time_prc`

规则：

1. UTC 是业务判断标准。
2. PRC / 北京时间只用于展示和排查。
3. `open_time_ms` 或 `open_time_utc` 用于排序和连续性判断。
4. PRC 字段不得作为业务排序和连续性判断主依据。
5. UTC 与 PRC 转换必须通过统一时间工具。
6. 禁止在业务代码中到处手写 `+8 小时`。

---

### 6.3 OHLCV 字段

建议包含：

1. `open`
2. `high`
3. `low`
4. `close`
5. `volume`
6. `quote_volume`

说明：

1. `volume` 表示成交量。
2. `quote_volume` 表示成交额或成交额等价字段。
3. 字段精度必须满足 BTCUSDT U 本位合约行情分析需要。
4. 金额和价格字段不得使用容易丢失精度的浮点类型作为数据库存储主类型。

---

### 6.4 Binance 原始扩展字段

可根据 Binance REST K线字段保留：

1. `trade_count`
2. `taker_buy_volume`
3. `taker_buy_quote_volume`
4. `ignore_value`

说明：

1. `taker_buy_volume` 可用于后续分析主动买入量。
2. `taker_buy_quote_volume` 可用于后续分析主动买入成交额。
3. 当前阶段不一定使用这些字段做策略，但可以作为行情基础数据保留。
4. 字段含义必须在代码注释或说明文档中写清楚。

---

### 6.5 数据质量字段

建议包含：

1. `is_closed`
2. `is_valid`
3. `quality_status`
4. `created_at`
5. `updated_at`

规则：

1. 正式 K线表只允许写入已收盘 K线。
2. `is_closed` 应为 `true` 才能进入正式表。
3. 如有质量异常，应优先阻断正式写入。
4. 不应把质量明显异常的数据当作正常 K线写入正式表。
5. 质量检查详细结果应记录在 `data_quality_check` 或等价质量检查表中。

---

## 7. 唯一约束规则

每张周期 K线表都必须有唯一约束，避免重复写入。

唯一约束应至少包含：

1. `exchange`
2. `market_type`
3. `symbol`
4. `interval`
5. `open_time_ms`

或使用等价的 UTC 时间字段组合。

原则：

1. 同一交易所。
2. 同一市场类型。
3. 同一交易对。
4. 同一周期。
5. 同一开盘时间。

只能有一根正式 K线。

示例：

`BTCUSDT + 4h + 2026-05-06 04:00 UTC` 只能对应一条正式 4h K线。

---

## 8. 索引规则

K线表应围绕实际查询方式建立索引。

至少应考虑：

1. 按 `symbol + open_time_ms` 查询。
2. 按 `symbol + open_time_utc` 查询。
3. 按时间范围查询。
4. 按最近 N 根查询。
5. 按数据质量状态查询。
6. 按数据源查询。

`market_kline_4h` 的核心查询场景包括：

1. 获取最近 N 根 4h K线。
2. 获取某个时间范围内的 4h K线。
3. 检查最新一根 4h K线。
4. 检查某段时间是否连续。
5. 后续策略读取某个评估窗口内的 K线。
6. 后续复盘读取建议生成后的行情走势。

索引设计必须服务这些查询场景。

---

## 9. 不同周期之间的关联规则

不同周期 K线之间不得依赖自增 id 关联。

例如：

1. `market_kline_1m.id`
2. `market_kline_4h.id`
3. `market_kline_1d.id`

这些 id 只在各自表内表示数据库记录身份，不表示行情顺序，也不应用于跨周期关联。

不同周期之间应通过以下方式关联：

1. `symbol`
2. `open_time_ms`
3. `open_time_utc`
4. UTC 时间范围

示例：

某一根 4h K线：

1. `symbol = BTCUSDT`
2. `open_time_utc = 2026-05-06 04:00:00`
3. `close_time_utc = 2026-05-06 08:00:00`

它对应的 1m K线范围应通过时间区间查询：

1. `symbol = BTCUSDT`
2. `open_time_utc >= 2026-05-06 04:00:00`
3. `open_time_utc < 2026-05-06 08:00:00`

而不是通过 id 范围判断。

---

## 10. 与策略建议引用 K线的关系

未来策略建议必须能追溯到当时依据的 K线。

但策略建议引用 K线时，必须区分两个概念：

1. 数据库记录身份。
2. 行情时间顺序。

允许后续策略建议或证据表记录：

1. `kline_table`
2. `kline_id`
3. `symbol`
4. `interval`
5. `open_time_ms`
6. `open_time_utc`
7. `role`
8. `reason`

其中：

1. `kline_id` 可以用于定位具体数据库记录。
2. `open_time_ms` 和 `open_time_utc` 必须用于表达行情时间。
3. 不得用 `kline_id` 判断 K线先后顺序。
4. 不得用 `kline_id` 推断 K线是否连续。
5. 不得跨周期使用 id 做关联。

如果某条 K线已经被策略建议引用，后续发现 Binance 官方已收盘 K线与数据库已有记录不一致，不得静默覆盖历史数据。

应记录数据质量异常，并根据后续设计考虑：

1. K线变更历史记录。
2. K线修正记录。
3. 受影响建议标记。
4. 复盘时说明当时策略依据的数据版本。

---

## 11. Repository 边界

不同周期 K线应有清晰的 Repository 边界。

推荐：

1. `MarketKline4hRepository`
2. `MarketKline1mRepository`
3. `MarketKline1dRepository`

也可以在底层抽象通用 K线 Repository 能力，但业务入口必须保持周期边界清晰。

原则：

1. 4h 业务不应直接写 1m 表。
2. 1m 业务不应直接写 4h 表。
3. 1d 业务不应直接写 4h 表。
4. Repository 负责数据库读写。
5. Repository 不负责调用 Binance。
6. Repository 不负责调用 Hermes。
7. Repository 不负责调用大模型。
8. Repository 不负责策略判断。
9. Repository 不负责自动交易。

---

## 12. 质量检查边界

不同周期 K线应有独立或可明确区分的质量检查入口。

可以复用通用质量检查函数，但必须明确传入周期参数。

例如：

1. `interval = 1m`
2. `interval = 4h`
3. `interval = 1d`

质量检查必须基于该周期对应的时间间隔。

例如：

1. `1m` 相邻 K线间隔为 1 分钟。
2. `4h` 相邻 K线间隔为 4 小时。
3. `1d` 相邻 K线间隔为 1 天。

禁止：

1. 用 4h 的连续性规则检查 1m。
2. 用 1m 的连续性规则检查 4h。
3. 不传 interval 就做连续性判断。
4. 只依赖自增 id 判断连续性。
5. 只依赖行数判断连续性。

---

## 13. 采集代码复用边界

虽然 K线表按周期分表，但 REST 采集能力应尽量复用。

可以复用：

1. Binance REST 请求方法。
2. K线 parser。
3. 已收盘过滤。
4. 基础字段校验。
5. UTC / PRC 时间转换。
6. interval 毫秒数计算。
7. 连续性检查函数。
8. 错误包装。
9. 重试逻辑。

不得复用到混乱的地方：

1. 不得把 1m、4h、1d 的入库逻辑写在一个不可维护的大函数里。
2. 不得让一个函数同时决定所有周期的业务用途。
3. 不得让一个脚本无边界地采集所有周期。
4. 不得为了复用牺牲周期表边界。
5. 不得把不同周期的数据质量结果混成一类无法区分的记录。

正确方向是：

1. 通用能力复用。
2. 周期业务隔离。
3. 入库表隔离。
4. 质量检查对象可区分。
5. 入口脚本可区分。
6. 模块说明文件写清楚每个周期的数据流。

---

## 14. 当前阶段实现范围

当前阶段只要求优先实现：

1. `market_kline_4h`
2. 4h K线 ORM。
3. 4h K线 Repository。
4. 4h K线唯一约束。
5. 4h K线基础索引。
6. 4h K线历史回补。
7. 4h K线增量采集。
8. 4h K线质量检查。

当前阶段可以暂不实现：

1. `market_kline_1m`
2. `market_kline_1d`
3. 1m 采集任务。
4. 1d 采集任务。
5. 1m 质量检查入口。
6. 1d 质量检查入口。
7. 1m / 4h / 1d 联合分析。
8. 1m / 4h / 1d 策略联合使用。

但开发 4h 能力时，必须避免写死为“永远只有 4h”。

---

## 15. 与旧通用表的关系

如果项目早期曾经存在通用 `market_kline` 表，应明确其状态。

后续正式主周期数据应迁移到或直接写入 `market_kline_4h`。

规则：

1. 新开发的 4h 正式采集应写入 `market_kline_4h`。
2. 新开发的 4h 策略应读取 `market_kline_4h`。
3. 新开发的 4h 质量检查应检查 `market_kline_4h`。
4. 不应在新代码中继续把 4h 正式数据写入通用 `market_kline`。
5. 如果保留旧表，只能作为历史兼容或迁移过程使用。
6. Codex 不得在没有说明的情况下同时写入旧表和新表。

---

## 16. 对 Codex 的实现约束

Codex 实现相关代码时必须遵守：

1. 当前阶段优先实现 `market_kline_4h`。
2. 不得把 1m、4h、1d 正式数据混入同一张长期业务表。
3. 不得依赖自增 id 判断 K线顺序。
4. 不得依赖自增 id 关联不同周期 K线。
5. 不得用 id 范围推断某根 4h K线包含哪些 1m K线。
6. 不得用一张通用表模糊承载所有周期正式数据。
7. 不得为了省事把不同周期 Repository 混成不可维护的大类。
8. 不得为了省事把不同周期质量检查混成无法区分的结果。
9. 不得让 Repository 调用 Binance、Hermes 或大模型。
10. 不得在 4h 代码中提前实现 1m、1d 的完整业务。
11. 实现 4h 时应预留 interval 参数化和通用 parser 复用能力。
12. 实现 4h 时不得把系统写死到未来无法扩展 1m 和 1d。
13. 如果修改旧 `market_kline` 相关代码，必须说明原因和兼容方案。
14. 如果新增周期表，必须同步新增或调整对应 Repository、质量检查入口和模块说明文件。

---

## 17. 决策后果

本决策带来的结果：

1. 4h 主策略数据边界清晰。
2. 1m、4h、1d 不会混在一张难维护的大表中。
3. 1m 大数据量不会污染 4h 主策略查询。
4. 4h 数据质量检查可以更专注。
5. 后续策略引用 K线更清晰。
6. 后续复盘能更清楚地区分主周期和辅助周期。
7. 后续 Admin 展示更容易分周期组织。
8. 后续可以对不同周期采取不同索引、归档和优化策略。
9. 采集代码仍可复用，不会因为分表导致重复造轮子。
10. Codex 更容易理解不同周期的职责边界。
