# 0002-kline-source-and-time-rules.md

# 决策 0002：K线数据源与时间规则

## 1. 决策状态

已确定。

本决策用于约束系统中的 K线数据源、K线收盘判断、增量采集窗口、缺口补齐、时间字段、时区展示和行情排序规则。

后续数据采集、数据质量检查、策略分析、建议生命周期、复盘评估都必须遵守本决策。

---

## 2. 决策背景

本项目的主策略周期是 `4h`。

4h K线是后续策略信号、策略聚合、大模型分析、操作建议、建议复盘的核心数据基础。

如果 4h K线来源不稳定、时间规则混乱、未收盘 K线被错误写入、或者行情顺序依赖数据库自增 id，后续策略分析和复盘都会失真。

因此，系统必须提前明确：

1. 4h 主 K线以什么数据源为准。
2. WebSocket 数据能不能拼接 4h K线。
3. 什么样的 K线才允许写入正式 K线表。
4. 增量采集是否只拉最后一根 K线。
5. 如何发现漏采和缺口。
6. 如何区分重叠 K线和需要写入的新 K线。
7. 时间字段以哪个时区为业务标准。
8. 用户消息中如何展示时间。
9. 后续 1m、1d K线如何预留。

---

## 3. 决策内容

本项目确定以下规则：

1. 4h 主 K线以 Binance REST 返回的官方已收盘 K线为准。
2. WebSocket 不作为正式 4h 主 K线数据源。
3. WebSocket 不允许自行拼接正式 4h K线。
4. 系统内部业务判断统一使用 UTC。
5. PRC / 北京时间只用于用户展示、排查和阅读。
6. 行情排序和连续性判断必须基于 `open_time_ms`、`open_time_utc` 或等价 UTC 时间字段。
7. 禁止依赖数据库自增 id 判断 K线顺序。
8. 未收盘 K线不得写入正式 K线表。
9. 增量采集不得只拉取最后一根已收盘 K线。
10. 增量采集必须拉取最近多根 K线，用于重叠校验、连续性检查和缺口补齐。
11. 拉取多根 K线的目的不是整批重复写库，而是校验已有数据、发现缺口、补齐缺失数据。
12. K线采集失败或数据质量异常时，必须记录并通过 Hermes 提醒。
13. 数据质量异常时，不得盲目写入正式 K线表。

---

## 4. K线权威数据源

### 4.1 4h 主 K线数据源

4h 主 K线必须来自 Binance U 本位合约 REST K线接口返回的官方 K线。

系统应通过 REST 拉取 Binance 已生成的 K线数据，而不是自己根据 tick 或 WebSocket 数据拼接 4h K线。

原因：

1. Binance 官方 K线是交易所统一口径。
2. 自己拼接 K线容易受到断线、延迟、丢包、重连、时间边界误差影响。
3. 后续策略回测、复盘和实盘建议必须基于稳定一致的数据。
4. 官方 K线更容易与外部行情软件和交易所页面对照。
5. 自行拼接会增加数据质量风险和维护成本。

---

## 5. WebSocket 数据边界

WebSocket 可以用于：

1. 最新价格监控。
2. 10s 价格事件提醒。
3. Redis 最新价格缓存。
4. 快速波动提醒。
5. 后续提醒价格接近入场区、止盈区、止损区、减仓区、成本线等关键位置。

WebSocket 不得用于：

1. 拼接正式 4h 主 K线。
2. 替代 Binance REST 官方 4h K线。
3. 写入正式 4h K线表。
4. 作为策略主周期的权威数据源。
5. 因短时价格波动直接生成正式交易建议。
6. 自动下单、自动平仓、自动调仓。

如果后续需要使用 WebSocket 数据分析插针、快速波动或价格路径，只能作为辅助数据，不得替代官方已收盘 4h K线。

---

## 6. 已收盘 K线规则

正式 K线表只允许写入已收盘 K线。

判断原则：

1. K线必须已经完成对应周期。
2. 当前仍在形成中的 K线不得写入正式 K线表。
3. 策略分析不得使用未收盘 4h K线生成正式建议。
4. 数据质量检查必须区分已收盘 K线和未收盘 K线。
5. 历史回补和增量采集都必须过滤未收盘 K线。

对于 4h K线：

1. 只有当该 4h 周期已经结束，才允许作为正式 K线处理。
2. 系统可以拉取最近多根 K线，但必须过滤掉最后一根未收盘 K线。
3. 不得因为希望更早生成建议而提前使用未收盘 K线。

---

## 7. K线连续性规则

K线连续性判断必须基于时间字段，而不是数据库 id。

对于同一交易对、同一周期、同一数据源：

1. 相邻两根 4h K线的 `open_time_ms` 差值必须等于 4h 对应的毫秒数。
2. 如果中间缺少某根 K线，应判定为数据不连续。
3. 数据不连续时，不得继续盲目写入异常批次。
4. 数据不连续时，必须记录 `data_quality_check`。
5. 数据不连续时，必须记录 `collector_event_log`。
6. 数据不连续时，必须通过 Hermes 提醒用户。
7. 后续策略分析必须能识别数据质量状态。

示例：

如果数据库中已有：

1. `04:00 UTC`
2. `08:00 UTC`
3. `16:00 UTC`

但缺少：

1. `12:00 UTC`

则不能因为数据库 id 连续就认为行情连续。

行情连续性只能以 K线开盘时间判断。

---

## 8. 增量采集拉取窗口与缺口补齐规则

4h K线增量采集不得只拉取最后一根已收盘 K线。

系统每次执行增量采集时，应通过 Binance REST 拉取最近多根 4h K线，用于：

1. 校验 REST 返回数据本身是否连续。
2. 校验 REST 返回数据与数据库已有数据是否能衔接。
3. 发现数据库是否漏掉某些 Binance 官方已经存在的已收盘 K线。
4. 在数据质量确认无误后，补齐数据库中缺失的 K线。
5. 避免因为某一次任务失败、网络异常、服务器重启导致永久漏 K。

拉取多根 K线的目的不是把多根 K线全部重复写入数据库。

拉取多根 K线的目的是使用重叠窗口做连续性校验、已有数据一致性校验和缺口补齐。

---

### 8.1 重叠窗口原则

增量采集必须拉取包含数据库最新 K线附近重叠区间的数据。

示例一：

如果数据库中已有 4h K线：

1. `04:00 UTC`
2. `08:00 UTC`

下一次增量采集时，不应只拉取：

1. `12:00 UTC`

而应拉取一个包含重叠区间的窗口，例如：

1. `04:00 UTC`
2. `08:00 UTC`
3. `12:00 UTC`

系统应使用 `04:00 UTC` 和 `08:00 UTC` 校验 REST 返回数据与数据库已有数据是否一致，并确认 `12:00 UTC` 可以与 `08:00 UTC` 正常衔接。

最终实际需要写入数据库的只有：

1. `12:00 UTC`

已有的：

1. `04:00 UTC`
2. `08:00 UTC`

只用于校验，不应作为新的 K线重复插入。

---

### 8.2 漏采后的补齐规则

示例二：

如果数据库中已有 4h K线：

1. `04:00 UTC`
2. `08:00 UTC`

但 `12:00 UTC` 这一轮采集时 REST 请求失败，导致数据库没有写入 `12:00 UTC`。

到下一轮 `16:00 UTC` 采集时，系统不应只写入：

1. `16:00 UTC`

否则 `12:00 UTC` 会永久缺失。

系统应拉取一个包含重叠区间和缺失区间的窗口，例如：

1. `04:00 UTC`
2. `08:00 UTC`
3. `12:00 UTC`
4. `16:00 UTC`

系统应检查：

1. REST 返回的 `04:00 UTC`、`08:00 UTC`、`12:00 UTC`、`16:00 UTC` 是否连续。
2. REST 返回的 `04:00 UTC`、`08:00 UTC` 是否与数据库已有记录一致。
3. 数据库是否缺少 `12:00 UTC`。
4. `12:00 UTC` 是否可以由 REST 官方已收盘 K线补齐。
5. `16:00 UTC` 是否可以在补齐 `12:00 UTC` 后正常衔接。

如果校验通过，最终实际写入数据库的应该是：

1. `12:00 UTC`
2. `16:00 UTC`

不应重复写入：

1. `04:00 UTC`
2. `08:00 UTC`

也不应只写入：

1. `16:00 UTC`

因为只写入 `16:00 UTC` 会导致数据库中永久缺少 `12:00 UTC`，后续策略分析和复盘都会失真。

---

### 8.3 实际写入集合规则

增量采集应先区分三类 K线：

1. 数据库已存在的重叠 K线。
2. 数据库缺失但 REST 返回中存在的已收盘 K线。
3. REST 返回中未收盘或质量异常的 K线。

处理规则：

1. 数据库已存在的重叠 K线，只用于校验。
2. 数据库缺失但 REST 返回中存在、且质量检查通过的已收盘 K线，可以写入。
3. REST 返回中未收盘的 K线，必须过滤，不得写入正式 K线表。
4. REST 返回中质量异常的 K线，不得写入正式 K线表。
5. 如果数据库存在缺口，且本次 REST 返回数据可以完整补齐该缺口，则可以按时间顺序补齐缺失 K线。
6. 如果数据库存在缺口，但本次 REST 返回数据不能确认或不能补齐该缺口，则不得跳过缺口直接写入后续 K线。
7. 如果重叠 K线与数据库已有记录不一致，必须记录数据质量异常，并通过 Hermes 提醒用户，不得静默覆盖。

实际写入集合应理解为：

    REST 返回的已收盘 K线
    减去数据库已存在且校验一致的重叠 K线
    减去未收盘 K线
    减去质量异常 K线
    等于本次允许写入的缺失 K线

数据库唯一索引和 upsert 只能作为幂等兜底。

业务逻辑不能依赖“反正数据库会去重”来粗暴写入整批 REST 返回数据。

---

### 8.4 重叠 K线一致性校验

对于 REST 返回中已经存在于数据库的重叠 K线，系统应进行一致性校验。

至少应校验：

1. `open_time_ms`
2. `open_time_utc`
3. `close_time_ms`
4. `open`
5. `high`
6. `low`
7. `close`
8. `volume`
9. `quote_volume`

如果 REST 返回的官方已收盘 K线与数据库已有 K线不一致，系统不得静默覆盖数据库数据。

应处理为数据质量异常：

1. 写入 `data_quality_check`。
2. 写入 `collector_event_log`。
3. 通过 Hermes 提醒用户。
4. 根据后续设计决定是否需要专门的 K线变更历史记录。
5. 在没有明确变更策略前，不得直接覆盖历史 K线。

---

### 8.5 拉取数量规则

具体拉取数量不应在代码中写死。

推荐使用配置项控制，例如：

1. `KLINE_4H_INCREMENTAL_FETCH_LIMIT`
2. `KLINE_INCREMENTAL_OVERLAP_LIMIT`

默认规则：

1. 每次至少拉取最近 3 根已收盘 4h K线。
2. 如果需要更强容错能力，可以配置为最近 5 根或更多。
3. 拉取窗口必须覆盖数据库最新 K线附近的重叠区间。
4. 拉取窗口应能覆盖短时间任务失败造成的漏采。
5. 所有返回 K线仍必须过滤未收盘 K线。

如果数据库最新 K线距离当前时间过远，说明可能存在较大缺口。

此时不应靠普通增量采集强行处理全部缺口。

应转入历史回补流程，或触发专门的缺口修复流程，并通过 Hermes 提醒用户。

---

### 8.6 禁止行为

禁止：

1. 增量采集只拉取最后一根 K线后直接写库。
2. 增量采集拉取多根 K线后不做区分，直接整批写库。
3. 发现数据库缺少中间 K线时，只写最新 K线。
4. 发现缺口后跳过缺口继续写后续 K线。
5. 发现重叠 K线与数据库不一致时静默覆盖。
6. 依赖数据库自增 id 判断 K线连续性。
7. 依赖唯一索引代替业务层缺口判断。
8. 依赖 upsert 掩盖采集逻辑不清晰的问题。

---

## 9. 重复 K线与幂等规则

系统必须允许历史回补和增量采集重复执行。

重复执行时，不得产生重复 K线。

正式 K线表应通过唯一约束保证幂等。

唯一约束应至少包含：

1. `exchange`
2. `market_type`
3. `symbol`
4. `interval`
5. `open_time_ms`

或在周期专用表中使用等价唯一约束。

规则：

1. 历史回补重复执行，不应插入重复 K线。
2. 增量采集重复执行，不应插入重复 K线。
3. upsert 必须基于唯一键。
4. 不得依赖数据库自增 id 判断是否重复。
5. 不得通过删除重建的方式掩盖重复问题。
6. 唯一索引是最后兜底，不是替代业务判断的理由。
7. 增量采集应先判断哪些 K线是已有重叠数据，哪些 K线是缺失数据，再决定写入集合。

---

## 10. 时间规则

系统内部业务判断统一使用 UTC。

UTC 用于：

1. K线开盘时间。
2. K线收盘时间。
3. K线连续性判断。
4. K线排序。
5. 策略分析。
6. 建议编号。
7. 建议生命周期。
8. 复盘到期时间。
9. 数据质量检查。
10. 数据库中用于业务判断的时间字段。

PRC / 北京时间用于：

1. 用户阅读。
2. Hermes 微信提醒展示。
3. Admin 页面展示。
4. 人工排查。
5. 用户复盘时理解时间。
6. 用户手动执行记录时辅助查看。

核心规则：

1. 业务判断以 UTC 为准。
2. 展示可以使用 PRC / 北京时间。
3. 如果数据库保留 PRC 字段，该字段只用于展示和排查。
4. PRC 字段不得作为 K线排序、连续性判断、策略判断的主依据。
5. UTC 与 PRC 转换必须通过统一时间工具。
6. 禁止在业务代码中到处手写 `+8 小时`。

---

## 11. 用户消息中的时间展示规则

所有面向用户的消息中，如果出现时间，必须明确标注时区。

用户消息包括：

1. Hermes 微信提醒。
2. 策略建议消息。
3. 复盘消息。
4. 数据质量异常消息。
5. Admin 展示。
6. 人工执行记录说明。

推荐同时展示：

1. 北京时间。
2. UTC。

推荐格式：

1. 北京时间：2026-05-06 12:00:00
2. UTC：2026-05-06 04:00:00

禁止只写没有时区标识的时间，例如：

1. 04:00
2. 12:00
3. 2026-05-06 04:00

禁止单独使用 `CST` 表示北京时间，因为 `CST` 存在歧义。

推荐使用：

1. 北京时间
2. UTC+8
3. PRC

如果消息中展示策略建议编号，必须说明编号中的小时使用 UTC。

示例：

建议编号：`20260506-BTCUSDT-04`

说明：`04` 表示 UTC 04:00，对应北京时间 12:00。

---

## 12. 数据库时间字段规则

数据库中用于业务判断的时间字段必须采用 UTC。

K线表应至少保留以下时间表达中的一种或多种：

1. `open_time_ms`
2. `open_time_utc`
3. `close_time_ms`
4. `close_time_utc`

如保留 PRC 字段，可命名为：

1. `open_time_prc`
2. `close_time_prc`

PRC 字段用途：

1. 用户阅读。
2. 数据排查。
3. Admin 展示。
4. 人工核对。

PRC 字段不得用于：

1. K线连续性判断。
2. K线排序主依据。
3. 策略分析主依据。
4. 建议编号主依据。
5. 复盘窗口计算主依据。

---

## 13. 1m、4h、1d K线扩展规则

当前阶段优先实现 4h K线。

但开发 4h REST K线采集能力时，必须提前考虑 1m 和 1d 的复用空间。

从 Binance REST 接口角度看：

1. 1m K线。
2. 4h K线。
3. 1d K线。

它们本质上主要是 `interval`、`limit`、`start_time`、`end_time` 等参数不同。

因此，以下能力应尽量设计成可复用：

1. REST 请求。
2. K线解析。
3. 已收盘过滤。
4. 基础字段校验。
5. 连续性检查。
6. 时间转换。
7. 错误处理。

但不同周期的以下内容必须保持清晰边界：

1. 正式数据表。
2. Repository。
3. 质量检查入口。
4. 业务用途。
5. 策略使用方式。
6. 复盘使用方式。

周期用途：

1. 4h 是当前主策略周期。
2. 1m 后续主要用于复盘价格路径、插针、快速波动和细节行情。
3. 1d 后续主要用于大级别市场环境判断。

禁止为了复用而把 1m、4h、1d 的入库逻辑、质量检查结果和业务语义混成一个不可拆分的大函数。

---

## 14. 数据采集失败处理规则

如果 K线采集失败，必须处理为高级别异常。

采集失败包括：

1. Binance REST 请求失败。
2. Binance REST 超时。
3. Binance 返回异常。
4. 网络异常。
5. 解析失败。
6. 数据缺失。
7. 数据不连续。
8. 数据字段异常。
9. MySQL 写入失败。
10. 程序运行异常。

处理规则：

1. 必须记录 `collector_event_log`。
2. 必须记录必要的数据质量结果。
3. 必须通过 Hermes 提醒用户。
4. 不得只写日志。
5. 不得静默忽略。
6. 不得在数据质量失败时继续写正式 K线表。
7. 不得调用大模型解释基础采集异常。
8. 不得因为采集异常自动交易。

---

## 15. 对策略层的影响

未来策略层必须读取数据质量状态。

如果 K线数据不完整、不连续或质量异常：

1. 策略层不得忽略。
2. 策略层不得基于异常数据生成强操作建议。
3. 策略聚合层应将数据质量异常作为风控条件。
4. 最终建议可以降级为 `wait` 或 `stop_trading`。
5. 系统必须明确告诉用户是数据质量问题导致等待或停止交易。

数据质量异常不是普通日志问题，而是会影响策略判断的核心风险。

---

## 16. 对 Codex 的实现约束

Codex 实现相关代码时必须遵守：

1. 4h 主 K线只能使用 Binance REST 官方已收盘 K线。
2. 不得使用 WebSocket 拼接正式 4h K线。
3. 不得把未收盘 K线写入正式 K线表。
4. 不得用数据库自增 id 判断行情顺序。
5. 不得在业务代码中手写 `+8 小时`。
6. 不得绕过统一时间工具。
7. 不得绕过数据质量检查。
8. 不得在数据质量失败时写正式 K线表。
9. 不得只写日志不发 Hermes 提醒。
10. 不得让基础采集异常调用大模型。
11. 不得为了 1m、1d 预留而扩大当前阶段开发范围。
12. 不得把不同周期 K线的业务语义混写在一个不可拆分的大函数中。
13. 增量采集必须拉取最近多根 K线，用于重叠校验、连续性检查和缺口补齐。
14. 增量采集不得只拉取最后一根已收盘 K线后直接写库。
15. 增量采集拉取多根 K线后，必须区分“已有重叠 K线”和“数据库缺失 K线”。
16. 已有重叠 K线只用于校验，不应作为新数据重复写入。
17. 如果数据库缺少中间 K线，而 REST 返回数据可以补齐，应按时间顺序补齐缺失 K线。
18. 如果数据库缺少中间 K线，不得跳过缺失 K线只写最新 K线。
19. 如果重叠 K线与数据库已有记录不一致，不得静默覆盖，必须记录数据质量异常并通过 Hermes 提醒。
20. 唯一索引和 upsert 只能作为幂等兜底，不能替代业务层连续性判断和缺口识别。

## 17. 决策后果

本决策带来的结果：

1. 4h 主 K线来源稳定。
2. 回测、策略、复盘和建议使用同一数据口径。
3. WebSocket 不承担不适合它承担的正式 K线职责。
4. 未收盘 K线不会污染正式数据。
5. 增量采集可以发现短期漏采。
6. 增量采集可以通过重叠窗口补齐缺失 K线。
7. 增量采集不会因为拉取多根 K线就重复写入已有数据。
8. 时间规则统一，减少 UTC / 北京时间混乱。
9. 用户消息中的时间更清楚。
10. 后续 1m、1d 可以复用采集能力，但不会破坏 4h 主周期边界。
11. 数据质量异常会被及时发现和提醒。
12. 后续策略建议可以追溯到可靠 K线数据。
